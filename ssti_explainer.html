Server-Side Template Injection (SSTI)

OWASP Top 10 · A03 Injection · CWE-94 — the injection that leads directly to Remote Code Execution

SSTI happens when user input is embedded into a template string and then rendered — instead of being passed safely as a variable. The engine evaluates the payload as code, giving the attacker access to the full runtime and often the OS.

Select a template engine and enter a payload. The vulnerable server concatenates your input into the template — the secure server passes it as a variable.

Engine fingerprinting — the differential probe

The same payload produces different output in different engines, revealing exactly which one is running.

Probe: {{7*'7'}} Jinja2 → 7777777 (Python string × int = repetition) Twig → 49 (PHP numeric coercion) Pebble → error (strict typing) Probe: ${7*7} FreeMarker / Velocity → 49 Probe: <%=7*7%> ERB (Ruby) → 49 Decision tree: {{7*7}} == 49? → Jinja2 or Twig or Pebble {{7*'7'}} == 7777777? → Jinja2 {{7*'7'}} == 49? → Twig ${7*7} == 49? → FreeMarker / Velocity <%=7*7%> == 49? → ERB

SSTI's most dangerous property: Python, Java, and Ruby template engines expose the full language runtime — including OS command execution. Here is the escalation path for each.

A systematic approach to finding SSTI across an application.

1
Find all reflected input surfaces
Any place the app echoes your input back: search bars, username fields, error messages, custom greetings, email templates, PDF generators, URL parameters. If you see your input in the response — test it.
2
Send a math probe first — simplest confirmation
Try {{7*7}}, ${7*7}, <%=7*7%>. If any returns 49 instead of the literal string, SSTI is confirmed. Math can't be XSS — this specifically tells you server-side execution is happening.
3
Distinguish SSTI from XSS
XSS executes in the browser. SSTI executes on the server. If {{7*7}} returns 49 in the HTTP response body — that's SSTI. If <script>alert(1)</script> triggers a browser alert — that's XSS. Very different exploits, very different severity.
4
Fingerprint the engine with differential probes
{{7*'7'}} returns 7777777 in Jinja2 (string repetition) but 49 in Twig (numeric coercion). This tells you exactly which exploit chain to use.
5
Test for blind SSTI via time delay
If output isn't reflected (email generation, background PDF), inject a sleep: {{range.constructor("return global.process.mainModule.require('child_process').execSync('sleep 5')")()}} — a 5-second response delay confirms execution with no visible output.
6
Automate with tplmap
tplmap is the dedicated SSTI scanner — analogous to sqlmap for SQL injection. It auto-detects the engine, confirms injection, and attempts RCE across 15+ engines automatically.
python tplmap.py -u 'http://target.com/page?name=*' # Detects engine, confirms RCE, offers interactive shell
① Never concatenate user input into template strings Root cause fix
Templates live in static files. User data is passed as context variables — never inserted into the template source string. This single rule prevents virtually all SSTI.
## Python / Jinja2 # Vulnerable template_str = "Hello " + request.args.get('name') env.from_string(template_str).render() # Safe template = env.get_template('greeting.html') # file contains: Hello {{ name }} template.render(name=request.args.get('name')) ## PHP / Twig # Vulnerable $twig->createTemplate("Hello " . $_GET['name'])->render([]); # Safe $twig->render('greeting.html.twig', ['name' => $_GET['name']]); ## Java / FreeMarker # Vulnerable Template t = new Template("name", "Hello " + userInput, cfg); # Safe Template t = cfg.getTemplate("greeting.ftl"); // file: Hello ${name} t.process(Map.of("name", userInput), out);
② Use logic-less / sandboxed engines Architectural
Logic-less engines like Mustache don't evaluate expressions — they only substitute variables. No expression language means no RCE surface.
# Mustache (Python) — no expression evaluation import pystache pystache.render("Hello {{name}}", {'name': user_input}) # Input {{7*7}} → "Hello {{7*7}}" — never evaluated # Jinja2 sandbox (partial — has been bypassed in the past) from jinja2.sandbox import SandboxedEnvironment env = SandboxedEnvironment() # Still: never concatenate user input into the template string
③ Validate and reject template syntax in input Defense in depth
Reject inputs containing template delimiters before they reach the rendering layer.
import re TEMPLATE_RE = re.compile(r'[{}<>%$#]|\$\{|<%|%>|\{\{|\}\}') def validate_name(user_input): if TEMPLATE_RE.search(user_input): raise ValueError("Invalid characters in input") return user_input # Or HTML-escape output so {{ becomes {{ from markupsafe import escape safe = escape(user_input)
④ Least-privilege process isolation Limit blast radius
Run the web process as a non-root user with minimal filesystem permissions. If RCE is achieved, the attacker is limited to what that account can access — no /etc/shadow reads, no system writes. Container isolation (Docker/K8s with read-only filesystem) adds another layer.
⑤ High-risk surfaces to audit
Email / PDF generators that embed user text  ·  Custom report templates editable by users  ·  Error pages that reflect URL parameters  ·  CMS "signature" or "custom greeting" fields  ·  Any feature saying "Hello [your name]" in a server-rendered page  ·  Notification templates where subject/body comes from user input
