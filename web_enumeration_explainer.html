Web Enumeration

Recon Phase · OWASP Testing Guide · The art of mapping what a target exposes before exploitation

Web enumeration is the systematic process of discovering everything a web application exposes — directories, files, subdomains, technologies, API endpoints, usernames, and misconfigurations — before attempting any exploitation. Every major vulnerability class is found faster with thorough enumeration.

Why enumeration comes before exploitation
You can't attack what you don't know exists
A hidden /admin panel or an undocumented /api/v2 route is invisible until enumerated. Most critical bugs hide in forgotten or unlinked endpoints.
Technology fingerprinting narrows the exploit list
Knowing it's WordPress 6.1 + WooCommerce on PHP 8.1 behind nginx immediately suggests which CVEs and misconfigs to test first.
Scope clarity prevents wasted effort
Enumeration reveals which assets are in scope, where CDN / WAF layers exist, and which IPs the application actually runs on.

Web enumeration follows a layered methodology — each phase builds on the last, expanding the attack surface map progressively.

1
Passive OSINT — zero interaction with target
Gather information from public sources without sending a single packet to the target. Completely undetectable.
## Google dorks site:target.com inurl:admin site:target.com ext:php OR ext:bak OR ext:sql site:target.com "index of" site:target.com filetype:env OR filetype:log "target.com" inurl:wp-admin ## Certificate transparency — find all subdomains https://crt.sh/?q=%.target.com curl "https://crt.sh/?q=%25.target.com&output=json" | jq '.[].name_value' ## Archive / history https://web.archive.org/web/*/target.com/* ## GitHub dorks for leaked secrets org:target-company "api_key" OR "secret_key" OR ".env" filename:.env "DB_PASSWORD" ## Shodan / Censys — exposed infrastructure shodan search "ssl.cert.subject.cn:target.com" censys search "target.com" --index=certificates
2
DNS enumeration — discover all subdomains
A forgotten subdomain — dev.target.com, staging.target.com, old.target.com — is often less hardened than the main site and runs older software.
## Passive (no direct DNS queries to target) subfinder -d target.com -all -recursive amass enum -passive -d target.com ## Active brute force gobuster dns -d target.com -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt ffuf -u https://FUZZ.target.com -w subdomains.txt -mc 200,301,302,403 ## Zone transfer attempt (rarely works but worth trying) dig axfr @ns1.target.com target.com ## Reverse DNS on IP ranges for i in $(seq 1 254); do host 10.10.1.$i | grep "domain name"; done ## Virtual host enumeration (same IP, different Host headers) gobuster vhost -u http://target.com -w subdomains.txt
3
Port & service discovery
Web apps often run on non-standard ports. 8080, 8443, 8888, 4443 are common. An admin interface on port 8082 might have no authentication.
## Quick top-ports scan nmap -sV -sC --top-ports 1000 target.com ## Full port scan (all 65535) nmap -p- -T4 target.com ## Web-specific service detection nmap -sV -p 80,443,8080,8443,8888,9000,9090,3000,5000 target.com ## Check for HTTP on every open port for port in 80 443 8080 8443 8888 9000 3000 5000; do curl -sk --max-time 3 http://target.com:$port | grep -i "<title>" done
4
Technology fingerprinting
Identify exactly what the target runs — CMS, framework, server, CDN, WAF — to map directly to known CVEs and default misconfigurations.
## Wappalyzer / WhatWeb whatweb -a 3 https://target.com wappalyzer https://target.com # browser extension or CLI ## Response headers reveal the stack curl -sI https://target.com | grep -iE "server:|x-powered-by:|x-generator:|via:" # Server: Apache/2.4.51 → known CVE-2021-42013 # X-Powered-By: PHP/7.4.3 → outdated PHP # X-Generator: Drupal 9 → check Drupageddon ## robots.txt and sitemap curl https://target.com/robots.txt curl https://target.com/sitemap.xml ## CMS detection # WordPress: /wp-login.php, /wp-json/wp/v2/users # Drupal: /CHANGELOG.txt, /core/CHANGELOG.txt # Joomla: /administrator/, /README.txt # Laravel: /_ignition/health-check, /api/user
5
Directory & file brute force
Discover hidden paths, backup files, admin panels, and exposed configuration files by brute-forcing against common wordlists.
## gobuster gobuster dir -u https://target.com -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories.txt -x php,html,txt,bak,zip,sql -t 50 ## ffuf ffuf -u https://target.com/FUZZ -w common.txt -mc 200,301,302,401,403 -c ## feroxbuster (recursive) feroxbuster -u https://target.com -w dirs.txt --depth 3 -x php,bak,txt ## High-value paths to always check /.git/HEAD ← git repo exposed /.env ← credentials /backup.zip ← full source code /phpinfo.php ← PHP config disclosure /server-status ← Apache mod_status /.htpasswd ← password file /swagger.json ← full API spec /graphql ← GraphQL endpoint /actuator ← Spring Boot metrics
6
JavaScript & API endpoint extraction
Modern SPAs embed their entire API surface in JavaScript bundles. Mining JS files reveals endpoints, authentication flows, and sometimes hardcoded secrets.
## Extract all JS files katana -u https://target.com -js-crawl -o jsfiles.txt gau target.com | grep "\.js$" ## Find endpoints in JS bundles python3 LinkFinder.py -i https://target.com/app.js -o cli ## Find secrets / keys in JS trufflehog filesystem --directory /path/to/js/ grep -rE "(api_key|secret|password|token|auth)['\"]?\s*[=:]\s*['\"][a-zA-Z0-9_-]{10,}" *.js ## Check source maps (sometimes expose original source) curl https://target.com/static/js/main.chunk.js.map ## GraphQL introspection curl -s -X POST https://target.com/graphql \ -H "Content-Type: application/json" \ -d '{"query":"{__schema{types{name fields{name}}}}"}'

403 bypass techniques

A 403 means the path exists but access is restricted — often by a WAF or misconfigured rule that can be bypassed.

## Path manipulation /admin/ → /.//admin// /admin → /ADMIN (case folding) /admin → /admin/. ## Header-based bypass X-Original-URL: /admin X-Rewrite-URL: /admin X-Forwarded-For: 127.0.0.1 X-Custom-IP-Authorization: 127.0.0.1 ## HTTP verb confusion GET /admin → 403 POST /admin → 200 (different verb, different ACL) ## URL encoding /admin → /%61dmin (URL-encoded 'a') /admin → /admin%20 (trailing space)

Technology fingerprinting maps the exact software stack — server, language, framework, CMS, libraries, CDN, WAF. Every identified component leads directly to known CVEs and default misconfigurations.

HTTP headers — the most revealing source
## Headers that leak technology Server: Apache/2.4.51 (Ubuntu) → Apache 2.4.51 → CVE-2021-42013 (RCE) X-Powered-By: PHP/7.4.3 → PHP 7.4.3 EOL → unpatched vulns X-Generator: Drupal 9 (https://www.drupal.org) → Check SA-CORE-2018-002 (Drupalgeddon2) X-AspNet-Version: 4.0.30319 → .NET version → ViewState exploits Set-Cookie: PHPSESSID=... → PHP Set-Cookie: JSESSIONID=... → Java/Tomcat Set-Cookie: ASP.NET_SessionId=... → .NET Via: 1.1 varnish → Varnish cache CF-RAY: ... → Cloudflare CDN X-Cache: HIT from squid → Squid proxy
CMS-specific fingerprints
## WordPress /wp-login.php → admin login /wp-json/wp/v2/users → user enumeration! /wp-content/ → plugin list /xmlrpc.php → brute force vector Generator meta tag: WordPress 6.1.1 ## Drupal /CHANGELOG.txt → exact version /core/CHANGELOG.txt Generator: Drupal 9.x ## Joomla /administrator/ /README.txt → version /configuration.php~ → backup ## Laravel /_ignition/health-check /.env exposed → common misconfiguration /storage/logs/laravel.log ## Django /admin/ → Django admin DEBUG=True → full stack traces
WAF detection — know before you probe

Detecting a WAF lets you choose the right bypass technique. Sending unfiltered payloads against a WAF wastes time and gets you blocked.

## wafw00f — automated WAF detection wafw00f https://target.com # Detected: Cloudflare [Generic] # Detected: AWS WAF [Amazon] # Detected: ModSecurity [Apache] ## Manual WAF fingerprinting # Send an obvious XSS payload and check the response: curl -s "https://target.com/?q=<script>alert(1)</script>" # 403 + "Access Denied" → WAF present # Cloudflare: "Attention Required! | Cloudflare" # AWS WAF: 403 with request ID header # Akamai: 403 with "Reference #" in body ## Common WAF bypass starters # URL encoding: %3Cscript%3E # Case variation: <ScRiPt> # HTML entities: <script>
Version → CVE mapping
## After fingerprinting, immediately check: searchsploit "Apache 2.4" searchsploit "WordPress 6.1" ## NVD lookup https://nvd.nist.gov/vuln/search?query=wordpress+6.1 ## PoC databases https://www.exploit-db.com/?q=drupal+9 https://packetstormsecurity.com/ ## Nuclei — automated CVE templates nuclei -u https://target.com -t cves/ -severity critical,high ## WPScan for WordPress wpscan --url https://target.com --enumerate vp,vt,u # vp = vulnerable plugins, vt = vulnerable themes, u = users

The core toolchain used in professional web enumeration assessments. Each tool excels at a specific phase.

gobuster
Dir / DNS / VHost
Fast Go-based brute forcer for directories, DNS subdomains, and virtual hosts. Concurrent, simple, reliable.
gobuster dir -u https://target.com \ -w /usr/share/seclists/Discovery/Web-Content/raft-large-directories.txt \ -x php,html,txt,bak,zip,sql,conf \ -t 50 -o results.txt gobuster dns -d target.com \ -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-20000.txt \ -t 100
ffuf
Fuzzer
Extremely fast web fuzzer with flexible FUZZ keyword placement — directories, parameters, headers, POST bodies.
## Directory fuzzing ffuf -u https://target.com/FUZZ -w dirs.txt -mc 200,301,302,401,403 ## Parameter discovery ffuf -u https://target.com/page?FUZZ=test -w params.txt -mc 200 ## Subdomain fuzzing ffuf -u https://FUZZ.target.com -w subdomains.txt -mc 200,301
amass / subfinder
Subdomain
Passive + active subdomain enumeration using certificate transparency, DNS brute force, and 100+ data sources.
subfinder -d target.com -all -recursive -o subs.txt amass enum -active -d target.com \ -brute -w subdomains.txt \ -o amass-results.txt ## Combine and deduplicate cat subs.txt amass-results.txt | sort -u | httpx -mc 200,301,302
feroxbuster
Recursive
Recursive content discovery — automatically re-runs on every discovered directory, finding deeply nested paths.
feroxbuster -u https://target.com \ -w /usr/share/seclists/Discovery/Web-Content/common.txt \ --depth 3 \ -x php,bak,txt,conf,xml \ --auto-tune \ -o ferox-results.txt
nikto
Vuln scan
Web server scanner checking for 6700+ dangerous files, outdated versions, and common misconfigurations. Noisy but thorough.
nikto -h https://target.com -o nikto-results.txt -Format htm # With authentication nikto -h https://target.com -id admin:password # Tune for specific checks nikto -h target.com -Tuning 1234567890ab
nuclei
CVE / Vuln
Template-based vulnerability scanner with 5000+ CVE templates. Runs after fingerprinting to match known issues.
## Run all critical/high CVE templates nuclei -u https://target.com -t cves/ -severity critical,high ## Technology-specific nuclei -u https://target.com -t technologies/wordpress.yaml ## Expose common misconfigs nuclei -u https://target.com -t exposures/ -t misconfiguration/
katana / gau
Crawl / JS
katana crawls with JS rendering; gau fetches all known URLs from Wayback, CommonCrawl, URLScan. Both mine endpoints from JS bundles.
## Active crawl with JS execution katana -u https://target.com -js-crawl -depth 3 -o crawl.txt ## Passive URL collection gau --threads 5 target.com | tee gau-results.txt ## Extract endpoints from JS cat gau-results.txt | grep "\.js$" | \ xargs -I{} python3 LinkFinder.py -i {} -o cli 2>/dev/null
whatweb / wappalyzer
Fingerprint
Technology detection from response headers, body patterns, cookies, and script tags. Maps to version + CVE lookup.
whatweb -a 3 https://target.com -v # Aggressive scan (more requests) whatweb -a 4 https://target.com --log-json=tech.json # With authentication whatweb -a 3 --user admin --password secret https://target.com/admin

Operational security during enumeration — how to stay under the radar, avoid detection, and enumerate ethically.

Rate limiting & detection avoidance
## Slow down scans to avoid WAF rate limits gobuster dir -u https://target.com -w dirs.txt -t 10 --delay 100ms ## Randomize user agent ffuf -u https://target.com/FUZZ -w dirs.txt \ -H "User-Agent: Mozilla/5.0 (compatible; Googlebot/2.1)" ## Use a proxy / VPN rotation proxychains gobuster dir -u https://target.com -w dirs.txt ## Limit requests per second feroxbuster -u https://target.com --rate-limit 50 ## Spread scanning over time # Use --scan-limit or --time-limit flags
What leaves traces in server logs
High noise Full port scans (nmap), nikto, large directory brute force with default user agents, thousands of 404s in seconds
Medium noise Directory fuzzing at moderate speed, subdomain brute force, crawler traffic
Low noise Passive OSINT (crt.sh, Shodan), viewing robots.txt/sitemap.xml, curl for single files, Google dorks
Legal & ethical boundaries
✓
Always have written authorization — a bug bounty scope document, a pentest SOW, or explicit written permission before touching any target. Unauthorized enumeration is a crime in most jurisdictions regardless of intent.
✓
Verify scope — confirm which domains, IPs, and subdomains are in scope. An enumeration tool that discovers out-of-scope assets can probe them automatically — check every finding against the scope document.
✓
Don't cause damage — avoid scans that send requests to production databases, trigger alert storms, or consume excessive bandwidth. Prefer passive recon and low-speed active scans in production environments.
Recommended enumeration order (stealth-first)
1. Passive OSINT first (zero server contact) crt.sh → subfinder → Shodan → GitHub → Wayback 2. Single low-noise requests robots.txt → sitemap.xml → /favicon.ico hash → headers 3. Technology fingerprinting (1-2 requests) whatweb, wappalyzer, single curl 4. Targeted dir scan (high-value paths only) .git/HEAD .env backup.zip phpinfo.php — not full wordlist 5. Full wordlist scan (if authorized, during low-traffic hours) gobuster / ffuf with rate limiting 6. Active subdomain brute force Only after passive subdomain enum is exhausted 7. Crawler / JS mining katana with depth limit, avoid login flows
