Insecure Direct Object Reference (IDOR)

OWASP Top 10 · Broken Access Control · CWE-639

IDOR happens when an app exposes a raw internal object reference (ID, filename, key) and trusts the user to only request their own data — without verifying ownership on the server.

You are logged in as Alice (user_id: 1001). Request different invoice IDs to see how a vulnerable vs. secure server responds.

Request URL
GET /api/invoice?id=
❌ Vulnerable server — no ownership check
✅ Secure server — enforces ownership
① Numeric ID in URLMost common

Sequential integers are trivially enumerable — increment by 1 to walk all records.

GET /api/orders/1042 → attacker's order GET /api/orders/1043 → victim's order ← IDOR! GET /api/profile?user_id=99 → victim's profile
② UUID — still vulnerable without ownership checkOften missed

UUIDs prevent enumeration but not IDOR. If Alice gets Bob's UUID via a referral link or API leak, she can still access his record.

GET /api/doc/3f2d9a1b-4e82-4c1a-a9d2-... → Bob's doc (UUID leaked)
③ File / path referenceMedium

Filename or path passed as a parameter — often combined with path traversal.

GET /download?file=invoice_alice.pdf GET /download?file=../invoice_bob.pdf ← Bob's file!
④ Hidden parameter / POST bodyOften overlooked

ID lives in a JSON/POST body, not the URL — missed by testers scanning only query strings.

POST /api/update-profile {"user_id": 99, "email": "attacker@evil.com"} ← tampered
⑤ Predictable hash / indirect referenceAdvanced

MD5/SHA1 of a numeric ID looks opaque but the input space is tiny and brute-forceable.

GET /report?token=c4ca4238a0b923820d ← MD5("1") — predictable!

A systematic approach used in bug bounty hunting and penetration testing.

1
Map every endpoint with an object reference
Proxy all traffic (Burp Suite). Flag every parameter that looks like an ID — numeric, UUID, filename, hash — in URLs, POST bodies, headers, and cookies.
2
Create two test accounts (A and B)
Generate objects with account A. Replay those requests authenticated as B — if B gets A's data, IDOR confirmed. Use Burp's "Match and Replace" to auto-swap session tokens.
3
Test all HTTP verbs, not just GET
GET may be locked but PUT/DELETE/PATCH on the same path may not. DELETE /api/posts/42 with another user's session is a classic miss.
4
Check response bodies for nested IDs
API responses often embed order.customer_id, document.owner_id — use these in follow-up requests to reach other users' objects.
5
Test vertical IDOR, not just horizontal
Horizontal = user A accessing user B's data. Vertical = regular user accessing admin resources. Try user_id=1, role_id=0, account_type=admin.
✓
Automate with Burp Autorize
Autorize replays every request with a low-privilege session token and flags responses that match the original — turning manual testing into a passive scan across the entire app.
① Server-side ownership check — always Essential
Every request for an object must verify the requesting user owns or has permission to access it. Never trust the ID alone.
// ❌ Vulnerable — trusts the ID app.get('/invoice/:id', async (req, res) => { const invoice = await db.Invoice.findById(req.params.id); res.json(invoice); // any authenticated user gets any record }); // ✅ Secure — ownership enforced in the query app.get('/invoice/:id', requireAuth, async (req, res) => { const invoice = await db.Invoice.findOne({ _id: req.params.id, owner: req.user.id // ← MUST match the session user }); if (!invoice) return res.status(403).json({ error: 'Forbidden' }); res.json(invoice); });
② Indirect object references Defense in depth
Map real IDs to session-scoped tokens server-side. The client never sees the real database ID — nothing to tamper with.
// Session token → real DB id (server-side map) const sessionMap = { 'tok_abc123': { realId: 1042, userId: 1001 }, }; app.get('/invoice/:token', requireAuth, (req, res) => { const entry = sessionMap[req.params.token]; if (!entry || entry.userId !== req.user.id) return res.status(403).json({ error: 'Forbidden' }); // tok_abc123 is meaningless to any other user });
③ UUIDs + ownership check Not UUIDs alone
UUIDs stop enumeration but do NOT prevent IDOR. A UUID is secrecy, not authorization. Always pair with an ownership check — if the UUID leaks via referral link, API log, or browser history, the record is exposed without a server-side check.
④ RBAC for vertical IDOR Good
Check both identity and role before serving any elevated resource.
function requireRole(role) { return (req, res, next) => { if (!req.user || req.user.role !== role) return res.status(403).json({ error: 'Forbidden' }); next(); }; } app.delete('/admin/users/:id', requireAuth, requireRole('admin'), deleteUserHandler);
⑤ Testing checklist
Add object-level auth to every code review checklist  ·  Use Burp Autorize in every pentest  ·  Log and alert on 403/401 spikes (enumeration signal)  ·  Run two-account IDOR tests in CI  ·  Prefer findOne({_id, owner}) over fetch-then-compare
