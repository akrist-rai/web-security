Subdomain Takeover

CWE-350 · Dangling DNS Records — claim a victim's subdomain by registering the deleted cloud resource it points to

Subdomain takeover happens when a DNS record — usually a CNAME — still points to a cloud service (GitHub Pages, Heroku, S3, Netlify, etc.) that no longer exists or was deleted, leaving the name unclaimed and available for anyone to register.

A records (not CNAME) — also vulnerable

Not all dangling records are CNAMEs. An A record pointing to a cloud IP (Elastic IP, Azure Public IP) is also vulnerable if the IP is released.

## AWS Elastic IP — released but DNS still points to it subdomain.corp.com A 54.23.45.67 ← old Elastic IP, now released ## Check if the IP is re-assignable: # AWS EIPs are re-assigned to other customers after release # Register a new AWS account → get lucky → receive that IP # → now own subdomain.corp.com via A record ## Cloud IP ranges (check if IP is in cloud range) curl https://ip-ranges.amazonaws.com/ip-ranges.json | jq '.prefixes[] | select(.ip_prefix | test("54.23.45.0/24"))'

A systematic workflow for finding subdomain takeovers — used in bug bounty programs and penetration tests.

1
Enumerate all subdomains
Build a complete list using passive sources (certificate transparency, DNS brute force, Wayback Machine). The wider the list, the more takeover candidates.
subfinder -d corp.com -all -o subdomains.txt amass enum -passive -d corp.com -o amass.txt cat subdomains.txt amass.txt | sort -u > all-subs.txt
2
Resolve all subdomains — find CNAMEs and IPs
Use dnsx or massdns to resolve every subdomain. Extract CNAME chains — the final CNAME target is the one to check against service fingerprints.
## dnsx — fast resolver with CNAME extraction dnsx -l all-subs.txt -cname -resp -o resolved.txt ## massdns — high-speed bulk resolution massdns -r resolvers.txt -t CNAME all-subs.txt > cnames.txt ## Extract just the CNAME targets cat resolved.txt | grep "CNAME" | awk '{print $NF}' | sort -u > cname-targets.txt
3
Check each CNAME target against service fingerprints
Fetch the response body of each CNAME target and match against known "unclaimed" fingerprints. This is what automated tools do at scale.
## subjack — automated takeover scanner subjack -w all-subs.txt -t 100 -timeout 30 -o results.txt -ssl ## subzy — checks 50+ services subzy run --targets all-subs.txt --concurrency 100 --output results.json ## nuclei with takeover templates nuclei -l all-subs.txt -t takeovers/ -c 50 -o nuclei-results.txt ## Manual check example: curl -sI https://corp-old-blog.github.io | head -5 # "There isn't a GitHub Pages site here" → dangling → takeover possible
4
Claim the dangling service (proof of concept)
Once a dangling CNAME is confirmed, register the service name. For responsible disclosure, serve a simple HTML page proving ownership — never deploy malicious content.
## GitHub Pages example # CNAME: corp-old-blog.github.io is dangling # 1. Create GitHub account / use existing # 2. Create repo named: corp-old-blog # 3. Enable GitHub Pages in repo settings # 4. Add CNAME file: echo "blog.corp.com" > CNAME # 5. Push an index.html with PoC content # Now blog.corp.com serves your GitHub Pages repo! # Proof: https://blog.corp.com → your PoC page ## S3 bucket example aws s3 mb s3://corp-assets --region us-east-1 # claim the bucket name echo '<h1>Subdomain Takeover PoC</h1>' | aws s3 cp - s3://corp-assets/index.html aws s3 website s3://corp-assets --index-document index.html
5
Monitor continuously — new subdomains appear daily
## Set up continuous monitoring with GitHub Actions or cron # Daily: re-run subfinder + subjack on the target domain # Alert on new CNAMEs pointing to unclaimed services # Tools: dnsreaper, monitored (GitHub), nuclei with cron

Subdomain takeover is not just defacement — owning a subdomain on a trusted domain unlocks a chain of high-severity attacks.

① Cookie theft — session hijackingCritical

Session cookies scoped to .corp.com (parent domain) are sent to all subdomains — including the attacker-controlled one. Any user who visits the taken-over subdomain leaks their session cookie.

## Cookie set by legitimate app: Set-Cookie: session=abc123; Domain=.corp.com; Secure; HttpOnly ## Victim visits blog.corp.com (attacker controls it) ## Browser sends: Cookie: session=abc123 ## Attacker's server logs the cookie → session hijacked ## Attacker also controls CSP of the subdomain ## Can set relaxed CSP → load scripts → steal even HttpOnly cookies ## via timing attacks or XSS gadgets
② TLS certificate issuance — valid HTTPSCritical

Let's Encrypt and other ACME CAs verify domain ownership by serving a challenge file at the domain. Since the attacker controls the subdomain, they can issue a valid TLS certificate — the green padlock shows blog.corp.com with no warning.

## Issue cert for the taken-over subdomain certbot certonly --webroot -w /var/www/html \ -d blog.corp.com # ACME challenge served from attacker's server → cert issued! # Users see: 🔒 blog.corp.com — no security warning
③ Email sending — trusted senderHigh

If the taken-over subdomain is an email subdomain (e.g. mail.corp.com or a SendGrid/Mailgun CNAME), the attacker can configure it to send emails appearing to come from @mail.corp.com — passing SPF checks.

## Dangling SendGrid subdomain: em123.corp.com → em123.sendgrid.net ## Register the SendGrid account → claim em123.sendgrid.net ## Now send emails "from" em123.corp.com → passes SPF!
④ CSP / CORS bypassHigh

If the main app has a CSP or CORS policy that allows *.corp.com, the attacker-controlled subdomain is a trusted origin — scripts loaded from it execute on the main domain.

## Main app CSP: Content-Security-Policy: script-src 'self' *.corp.com ## Attacker controls blog.corp.com ## Hosts malicious JS at: https://blog.corp.com/evil.js ## Any page on corp.com can now load this script with full trust! ## CORS: Access-Control-Allow-Origin: *.corp.com ## Attacker sends XHR from blog.corp.com → reads API responses
⑤ Phishing on trusted domainHigh

Host a convincing login page at login.corp.com or sso.corp.com. Users and security tools see a valid cert and the legitimate corporate domain — extremely effective phishing.

① Remove dangling DNS records immediately Root fix
When decommissioning any cloud service, delete the DNS record first, then delete the service. Never delete the service and leave the DNS record. Build this into your offboarding checklist.
## DNS cleanup checklist when deleting a service: 1. Identify all DNS records pointing to the service dig +short CNAME blog.corp.com 2. Delete the DNS record FIRST # Route53: aws route53 change-resource-record-sets --hosted-zone-id Z123 \ --change-batch '{"Changes":[{"Action":"DELETE","ResourceRecordSet":{...}}]}' 3. Wait for TTL to expire (confirm propagation) watch -n 5 "dig +short blog.corp.com" 4. Then delete the service (GitHub repo, S3 bucket, Heroku app) ## Order matters: DNS first, service second — always.
② Audit all DNS records regularly Ongoing
Export your entire DNS zone and check every CNAME/A record to confirm the target is still claimed and serving expected content. Automate this monthly at minimum.
## Export all DNS records (Route53) aws route53 list-resource-record-sets --hosted-zone-id Z123 \ | jq '.ResourceRecordSets[] | select(.Type == "CNAME") | {Name, Value: .ResourceRecords[0].Value}' ## Check each CNAME target while read cname; do response=$(curl -sk --max-time 5 "https://$cname" | head -c 500) echo "$cname: $response" done < cname-targets.txt ## Automated: dnsReaper pip install dnsreaper dnsreaper scan --provider aws --aws-profile default ## nuclei takeover templates on your own DNS nuclei -l your-subdomains.txt -t takeovers/ -o audit-results.txt
③ Use a DNS ownership verification token Prevention
Some services (Azure Static Web Apps, Vercel) require you to add a TXT record proving DNS ownership before a custom domain is accepted — preventing someone else from claiming it even if the CNAME is dangling.
## Azure Custom Domain verification token ## Before the service accepts your custom domain, add: _dnsauth.blog.corp.com TXT "abc123verificationtoken" ## This means even if the CNAME is dangling: ## attacker claims Azure service → Azure asks for TXT proof ## attacker can't add TXT to corp.com's DNS → claim rejected ## GitHub Pages: verify your organization domain ## Settings → Pages → Add a verified domain → adds TXT record ## Once verified, ONLY your org can use *.corp.com for Pages
④ Scope cookies to exact domain, not parent Defense in depth
If cookies use Domain=.corp.com, a taken-over subdomain receives them. Scope session cookies to the exact domain instead — Domain=app.corp.com — to limit blast radius.
⑤ Key takeaways
Always delete DNS before deleting the service  ·  Audit all CNAMEs quarterly with dnsReaper or nuclei  ·  Use services that require ownership TXT verification  ·  Scope cookies to exact domain — not *.corp.com  ·  Include DNS cleanup in all service decommission checklists  ·  Subscribe to certificate transparency logs to catch unauthorized cert issuance (crt.sh, Facebook CT monitor)
