MFA Bypass

Authentication Design Flaws · CWE-287 · Multi-factor authentication raises the bar — but implementation gaps let attackers step right over it

MFA adds a second factor — something you have (TOTP app, hardware key) or something you are (biometric) — to complement the password. It dramatically raises the cost of account takeover. But bad implementations introduce gaps that let attackers bypass the factor entirely, without touching the code itself.

MFA factor types — attack resistance spectrum
SMS OTP
SIM swap, SS7 hijack, phishing. Weakest factor.
Email OTP
Email account compromise. Better than SMS but still phishable.
TOTP (Google Auth)
Real-time phishing (AiTM), seed backup theft. Moderate resistance.
Push notification
MFA fatigue bombing. Strong when used correctly.
Hardware key (FIDO2)
Phishing-resistant by design — binds to origin. Strongest available.
Attack categories
● OTP brute force (no rate limit on code entry)
● Flow bypass (skip MFA step entirely via URL)
● Response manipulation (change "fail" to "success")
● Backup code abuse (weak codes, no rate limit)
● AiTM phishing (real-time session relay)
● MFA fatigue (push bombing)
● SIM swap (take over phone number)
● SS7 hijack (intercept SMS at telecom level)
① OTP brute force — no rate limitCritical

A 6-digit TOTP has 1,000,000 possible values. If there is no rate limit or lockout on the code entry endpoint, an attacker with a valid password can brute force it in under a second via automation.

# 6-digit SMS/TOTP: 000000–999999 = 1,000,000 values # With no rate limit: for code in $(seq -w 0 999999); do curl -s -X POST https://target.com/verify-mfa \ -d "code=$code&session=VICTIM_SESSION" | grep -i "success" done # GraphQL batching bypass (from API Security module): mutation { a1: verifyMFA(code: "000000") { success } a2: verifyMFA(code: "000001") { success } ...all 1M codes in one request... } # TOTP window: typically 30-second window, server often accepts # current AND previous code → effectively 2 valid codes at any time # HOTP (counter-based): no time window → longer attack window
② Backup codes — weak generation or no rate limitHigh
# Backup codes are often: # - Short (6-8 characters) vs TOTP's 6 digits # - Stored in plaintext in DB (leaked in breach) # - Not rate-limited (separate endpoint from main MFA) # - Reusable after "recovery" (some apps regenerate same codes) # Test: does the backup code endpoint have rate limiting? for code in AAAA1111 AAAA1112 AAAA1113 ...; do POST /account/recovery backup_code=$code done # Also: check if backup codes are displayed in HTML source, # stored in localStorage, or sent in a cookie
③ TOTP seed backup exposureHigh
# When setting up TOTP, the app shows a QR code containing the seed secret # Format: otpauth://totp/Service:user@example.com?secret=BASE32SECRET # If the secret is: # - Stored in plaintext in DB → leaked in breaches # - Transmitted without HTTPS # - Cached in browser after setup page # - Exposed in an IDOR (GET /api/users/OTHER_ID/totp-seed) # With the seed, attacker generates all past and future codes: import pyotp totp = pyotp.TOTP("STOLEN_BASE32_SECRET") print(totp.now()) # current valid code — always # MFA provides zero additional protection once seed is stolen

The most impactful MFA bypasses don't touch the OTP at all — they exploit logic flaws in the authentication flow itself.

① Direct endpoint access — skip MFA stepCritical

MFA creates a "pending authentication" state between password verification and code verification. If the app doesn't properly enforce that MFA was completed, navigating directly to the post-login page may work.

# Normal flow: # GET /login → POST /login (pass) → GET /mfa → POST /mfa (code) → GET /dashboard # Attack: after step 2 (password accepted), skip step 4: # 1. POST /login → success, session marked "password_verified" # 2. SKIP the MFA page entirely # 3. GET /dashboard directly # → If app only checks "password_verified" flag and not "mfa_verified": BYPASSED! # Also test: GET /account/settings (usually requires full auth — skippable?) GET /api/v1/profile (API might have weaker MFA enforcement than web UI) GET /dashboard?skip_mfa=true (parameter manipulation)
② Response manipulation — change false to trueHigh

If MFA verification trusts a client-side response, intercepting and modifying it with Burp can bypass the check entirely.

# Submit wrong MFA code — intercept the response in Burp: POST /verify-mfa code=000000 → HTTP 200: {"success": false, "message": "Invalid code"} # Modify response before browser processes it: {"success": true, "message": "Code verified"} ← changed! # If app redirects based on the JSON response body # rather than server-side session state → bypass! # Also: change HTTP status code # 401 → 200 in response → app treats as success # Proper implementation: server MUST update session state on its side # Client response is only informational — never trusted for access control
③ Password reset bypasses MFAHigh
# MFA on login, but password reset flow skips MFA check: # 1. Request password reset for victim's email # 2. Attacker has access to victim's email (phished, compromised) # 3. Click reset link → set new password # 4. Log in with new password → MFA required # 5. BUT: if victim has SMS MFA and attacker did SIM swap → bypass complete # Also test: # - Does password reset invalidate active sessions? (logout all devices) # - Does account recovery bypass MFA entirely? # - Does "trusted device" survive password reset? # "Remember this device" abuse: # If device trust token is predictable or not tied to account: # Attacker steals or guesses device token → MFA skipped on "trusted" device
④ Cross-account OTP usageMedium
# Attacker has their own account with MFA enabled # App doesn't bind OTP verification to a specific user session # Attack: # 1. Attacker logs in with victim's password (from breach/guess) # 2. MFA prompt appears for victim's account # 3. Attacker submits a valid OTP from THEIR OWN authenticator app # 4. If server only checks "is this a valid OTP in the current window?" # without checking "is this OTP for THIS user?" → bypass! # The OTP must be tied to: specific user + specific session + used only once

Technical MFA bypasses aside — social and infrastructure attacks often reach around MFA entirely.

AiTM (Adversary-in-the-Middle) phishing — real-time relayBypasses TOTP completely

Tools like Evilginx2 and Modlishka create a reverse proxy that sits between the victim and the real site. The victim enters their real password AND their real OTP — the proxy captures the session cookie instantly.

# Evilginx2 setup (educational overview): # 1. Point a phishing domain to Evilginx2 proxy # 2. Victim visits: accounts.google.phishing-domain.com # 3. Proxy relays to REAL accounts.google.com in real-time # 4. Victim sees legitimate Google login page # 5. Victim enters password → relayed to real Google → real MFA prompt returned # 6. Victim enters TOTP → relayed to real Google → session cookie issued # 7. Proxy captures the session cookie BEFORE passing it to victim # Result: attacker has a valid, authenticated session cookie # TOTP was correctly entered by the victim — it doesn't matter! # The session cookie is what was stolen # Countermeasure: ONLY FIDO2/WebAuthn hardware keys are phishing-resistant # WebAuthn binds the credential to the origin URL — phishing domain ≠ real domain # Even if victim enters password on phishing site, the key won't work there

# Attack on push-based MFA (Microsoft Authenticator, Duo Push): # 1. Attacker has valid credentials (from breach/phish) # 2. Attacker triggers login repeatedly → flood of push notifications to victim # 3. At 2 AM, victim wakes to 50 "Approve login?" notifications # 4. Victim accidentally or frustratedly taps "Approve" # 5. Attacker is in. # Real-world examples: # - Uber breach (2022): attacker pushed MFA for hours, then WhatsApp-messaged # victim claiming to be IT support → victim approved # - Rockstar Games breach (2022): similar technique # Mitigations: # - Number matching: push shows "Enter 42 from app" → defeats fat-finger approvals # - Additional context: push shows location + device → anomaly visible # - Rate limit push notifications (max 3 per 10 minutes) # - "Deny and report" button that triggers security alert
SIM swap & SS7 attacks — SMS MFA bypassHigh
# SIM swap: social engineering the carrier # Attacker calls carrier, impersonates victim → transfers number to attacker's SIM # All SMS to victim's number now go to attacker # → Receive SMS OTP → bypass MFA # Real targets: high-value accounts (crypto exchanges, executives) # Success rate: varies — some carriers have stronger verification # SS7 (Signaling System 7) attack: # Exploits decades-old telecom protocol vulnerabilities # Can intercept SMS without the carrier's cooperation # Requires access to SS7 network (telecom insider or nation-state) # Why SMS MFA is still better than nothing: # Raises attack cost significantly for most threats # But: for high-value targets, SMS MFA should be replaced with TOTP or FIDO2 # NIST SP 800-63B: SMS OTP is "restricted" — not recommended for new implementations # But: still far better than password-only!

① Rate limit and lockout on OTP entry Fixes brute force
The OTP endpoint needs the same — or stricter — rate limiting as the password endpoint. After 5 failures, lock the MFA session and require re-authentication.
## Express.js — OTP rate limiter const otpLimiter = rateLimit({ windowMs: 15 * 60 * 1000, // 15 minutes max: 5, // 5 attempts keyGenerator: (req) => req.session.userId || req.ip, handler: (req, res) => { // Invalidate the pending auth session req.session.pendingMFA = false; req.session.userId = null; res.status(429).json({ error: 'Too many attempts — please log in again' }); } }); app.post('/verify-mfa', otpLimiter, verifyMFAHandler); // Also: codes are single-use — mark as used after verification // Store used codes in cache for the TOTP window duration
② Enforce MFA completion server-side — explicit state machine Fixes flow bypass
Use an explicit three-state session model: anonymous → password_verified → fully_authenticated. Every protected endpoint checks for the fully_authenticated state. No skipping steps.
// Session states (server-side only — never trust client) const AUTH_STATES = { ANONYMOUS: 0, PASSWORD_VERIFIED: 1, // password OK, MFA pending FULLY_AUTHENTICATED: 2 // MFA complete — full access }; // Middleware — protect all routes function requireFullAuth(req, res, next) { if (req.session.authState !== AUTH_STATES.FULLY_AUTHENTICATED) { return res.status(401).redirect('/login'); } next(); } // Login flow app.post('/login', async (req, res) => { if (await checkPassword(req.body)) { req.session.authState = AUTH_STATES.PASSWORD_VERIFIED; req.session.pendingUserId = user.id; // NOT logged in yet res.redirect('/mfa'); } }); app.post('/verify-mfa', otpLimiter, (req, res) => { if (verifyOTP(req.body.code, req.session.pendingUserId)) { req.session.authState = AUTH_STATES.FULLY_AUTHENTICATED; req.session.userId = req.session.pendingUserId; res.redirect('/dashboard'); } });
③ Bind OTP to user and session — single use Fixes cross-account reuse
Verify the OTP against the specific user who is being authenticated, in the specific session. Mark it as used immediately to prevent replay.
## Python — TOTP verification with user binding import pyotp, redis def verify_totp(user_id, code, session_id): # Get user's specific TOTP secret (not any user's secret) secret = db.get_totp_secret(user_id) totp = pyotp.TOTP(secret) # Check single-use: has this code been used in this window? cache_key = f"used_otp:{user_id}:{totp.timecode(time.time())}" if redis.get(cache_key): raise ValueError("Code already used") if not totp.verify(code, valid_window=1): raise ValueError("Invalid code") # Mark as used for this time window redis.setex(cache_key, 60, "1") return True
④ Push MFA: enable number matching & context Fixes fatigue attacks
Number matching requires the user to enter a number shown on the login screen into the authenticator app — impossible to approve accidentally or without seeing the screen. Add location context so anomalies are visible.
## Microsoft Entra (Azure AD) — enable number matching # Admin center → Security → Authentication methods → Microsoft Authenticator # → Configure → Number matching: Enabled ## Additional context in push: # "Approve sign-in from Seattle, WA on Windows device?" # Victim seeing "Sign-in from Lagos, Nigeria" will likely deny ## Rate limit pushes: # Max 3 push notifications per 10-minute window # After 3 denials: require re-authentication with different factor ## "Report suspicious activity" button: # Deny + alert the security team immediately
⑤ FIDO2 / WebAuthn — phishing-resistant MFA Ultimate defence
WebAuthn hardware keys (YubiKey, passkeys) are cryptographically bound to the origin. An AiTM phishing proxy on a different domain cannot complete the WebAuthn flow — even if the victim is deceived.
## Registration (server-side, simplified) const options = await generateRegistrationOptions({ rpName: 'My App', rpID: 'app.example.com', // ← bound to this exact origin userID: user.id, userName: user.email, attestationType: 'none', authenticatorSelection: { residentKey: 'preferred', userVerification: 'preferred', }, }); ## Authentication const options = await generateAuthenticationOptions({ rpID: 'app.example.com' }); // Browser sends challenge to authenticator // Authenticator signs with private key ONLY IF rpID matches current origin // phishing-domain.com !== app.example.com → key refuses to sign → attack fails ## Libraries: @simplewebauthn/server (Node), python-fido2 (Python)
⑥ MFA security checklist
Rate limit OTP attempts — max 5 per 15 minutes per session  ·  Explicit server-side auth state machine — MFA completion checked on every request  ·  OTPs are single-use and bound to specific user + session  ·  No response-manipulation bypass — access decisions made server-side only  ·  Push MFA: enable number matching + context to defeat fatigue  ·  Replace SMS MFA with TOTP or FIDO2 for sensitive accounts  ·  Password reset flow must not bypass MFA  ·  "Remember device" tokens must be unpredictable, bound to user, and expiring  ·  For highest-security accounts: enforce FIDO2 hardware keys (phishing-resistant)
