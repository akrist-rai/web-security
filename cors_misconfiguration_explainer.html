① Use a strict allowlist — never reflect the Origin header Root fix
Maintain an explicit list of trusted origins. Only set ACAO to an exact match from the list. Never set ACAO to the value of the incoming Origin header without checking it.
## Node.js / Express const ALLOWED_ORIGINS = new Set([ 'https://app.bank.com', 'https://bank.com' ]); app.use((req, res, next) => { const origin = req.headers.origin; if (ALLOWED_ORIGINS.has(origin)) { res.header('Access-Control-Allow-Origin', origin); res.header('Access-Control-Allow-Credentials', 'true'); res.header('Vary', 'Origin'); // ← important for caching } // If not in allowlist: no CORS headers → browser blocks next(); }); ## Python / Django (django-cors-headers) CORS_ALLOWED_ORIGINS = [ "https://app.bank.com", "https://bank.com", ] CORS_ALLOW_CREDENTIALS = True # Never use CORS_ORIGIN_ALLOW_ALL = True with credentials ## Nginx if ($http_origin ~* "^https://(app\.)?bank\.com$") { add_header Access-Control-Allow-Origin $http_origin always; add_header Access-Control-Allow-Credentials true always; add_header Vary Origin always; }
② Never allow null origin in production Essential
The null origin is used by sandboxed iframes and local files — legitimate applications should never need to accept it. Remove it from any allowlist.
## Bad — never do this: if (origin === 'null' || !origin) { res.header('Access-Control-Allow-Origin', 'null'); res.header('Access-Control-Allow-Credentials', 'true'); } ## Good — simply don't include null in the allowlist const ALLOWED_ORIGINS = new Set(['https://app.bank.com']); // null will never match → no CORS headers set → blocked
③ Add the Vary: Origin header Required with dynamic ACAO
If ACAO varies per request (dynamic allowlist), you MUST set Vary: Origin. Without it, a CDN/proxy caches the first response with one ACAO value and serves it to all subsequent requests — potentially returning a response with the wrong origin allowed.
## Always include when ACAO is dynamic: res.header('Vary', 'Origin'); ## Example problem without Vary: # Request 1: Origin: https://app.bank.com → ACAO: https://app.bank.com (cached) # Request 2: Origin: https://evil.com → CDN serves cached response # → ACAO: https://app.bank.com in cached response # → Actually SAFE here, but logic is broken ## Another problem: CDN caches ACAO: evil.com (if misconfigured) # → All users receive that cached header → mass exploitation
④ Wildcard * — safe only for truly public data Know when it's OK
Access-Control-Allow-Origin: * is safe for public, unauthenticated endpoints (public CDN assets, public APIs with no user data). Never use wildcard with credentials or on authenticated endpoints.
## Safe use of wildcard: # Public font/asset CDN Access-Control-Allow-Origin: * # Public API returning non-user-specific data Access-Control-Allow-Origin: * ## NEVER: Access-Control-Allow-Origin: * Access-Control-Allow-Credentials: true ← browsers block this anyway, but signals bad intent ## NEVER on authenticated endpoints: GET /api/user/account → Access-Control-Allow-Origin: * ← wrong
⑤ Checklist
Maintain an explicit origin allowlist — never reflect the Origin header verbatim  ·  Use exact string matching — no substring, suffix, or regex match on origin  ·  Never add null to the allowlist  ·  Always set Vary: Origin when using a dynamic allowlist  ·  Never combine * with Allow-Credentials: true  ·  Audit all API endpoints — not just main app routes  ·  Third-party libraries may set permissive CORS defaults — always review
